🚨 Incident Response

O que fazer nos primeiros 5 minutos define o impacto do incidente. Incident Response é o processo estruturado para detectar, comunicar, diagnosticar, mitigar e resolver incidentes. Pânico piora tudo. Processo claro reduz o MTTR (Mean Time To Recovery).

💡 O Fluxo de Incident Response

Cada incidente segue este fluxo, independente da severidade:

  • •Detecção — Alerta automático (métricas, logs) ou report de usuário. Quanto mais rápido, menor o impacto
  • •Triagem e Severidade — SEV1 (sistema fora do ar) a SEV4 (cosmético). Define quem é acionado
  • •Incident Commander — Uma pessoa coordena. Delega diagnóstico, comunicação e mitigação
  • •Comunicação — Status page atualizada. Stakeholders informados. Canal dedicado (war room)
  • •Mitigação → Resolução — Primeiro para o sangramento (rollback, feature flag off), depois corrige a root cause

📊 Caso Real: PagerDuty Incident Response

O PagerDuty publicou seu playbook de incident response como open source:

  • Regra #1: O Incident Commander não debugga — coordena. Pessoas diferentes para diagnóstico e comunicação
  • Regra #2: Comunicação a cada 15 minutos, mesmo que não tenha novidade. Silêncio gera pânico
  • Regra #3: Mitigue antes de investigar. Rollback é sempre a primeira opção, mesmo se não sabe a causa
  • Resultado: MTTR médio de 25 minutos para SEV1, comparado com 2+ horas sem processo

💡 Dica Prática

Defina antes do incidente: quem é o Incident Commander, qual o canal de comunicação (Slack #incidents), como atualizar a status page, e qual o critério para cada nível de severidade. Quando o sistema cai às 3h da manhã, ninguém quer decidir processo — quer seguir processo.

📝 Post-Mortem

Sem post-mortem, o mesmo incidente acontece de novo. Post-mortem é o documento criado após cada incidente significativo: timeline, root cause, impacto e ações corretivas. O princípio fundamental é blameless: foca no sistema que permitiu o erro, não na pessoa que errou.

💡 Estrutura de um Post-Mortem Blameless

  • •Timeline — Reconstituição minuto a minuto: quando detectou, quem respondeu, o que fez, quando resolveu
  • •5 Whys — Pergunte “por quê?” 5 vezes para chegar na root cause real, não na superfície
  • •Impacto — Quantos usuários afetados, quanto tempo fora, impacto financeiro estimado
  • •Action Items — Ações concretas com owners e deadlines. “Melhorar monitoramento” não vale — “Adicionar alerta de latency p99 > 500ms até sexta, owner: João” vale

📊 Caso Real: Google SRE Post-Mortems

O Google publica post-mortems internos para que todos os times aprendam:

  • Blameless por design: Nunca “fulano errou”, sempre “o sistema permitiu que X acontecesse”
  • Template padronizado: “O que aconteceu” → “Por que aconteceu” → “O que vamos mudar” → “Action items”
  • Follow-up obrigatório: Action items são trackados como bugs. Se não foram feitos em 30 dias, escalam
  • Compartilhamento: Qualquer eng do Google pode ler qualquer post-mortem. Aprendizado transversal

✓ O que FAZER

  • ✓Conduzir post-mortem em até 48h após o incidente
  • ✓Incluir todos que participaram da resposta
  • ✓Trackear action items como tickets com deadline

✗ O que NÃO fazer

  • ✗Culpar indivíduos — culpe o sistema que permitiu o erro
  • ✗Escrever action items vagos como “melhorar testes”
  • ✗Pular o post-mortem porque “já resolveu”
3

📖 Runbooks

Em uma emergência às 3h da manhã, ninguém lembra o procedimento. Runbooks são documentação passo-a-passo para cenários comuns: “banco lento”, “serviço não responde”, “disco cheio”, “certificado expirado”. O runbook é o GPS do on-call — siga sequencialmente em vez de improvisar.

💡 Anatomia de um Bom Runbook

  • •Trigger — Que alerta/situação ativa este runbook
  • •Decision tree — Diagnóstico estruturado: “Se X, vá para passo 3. Se Y, vá para passo 5”
  • •Comandos copy-paste — Comandos exatos para executar, não descrições genéricas
  • •Escalation — Quando e para quem escalar se os passos não resolvem
  • •Living doc — Atualizado a cada uso. Se um passo não funcionou, corrija o runbook

📖 Exemplo: Runbook “Database Lento”

1

Check queries ativas

SELECT pid, query, state, age(clock_timestamp(), query_start) FROM pg_stat_activity WHERE state != 'idle' ORDER BY query_start;

2

Kill queries > 30s

SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state != 'idle' AND query_start < now() - interval '30 seconds';

3

Check locks e disk space

Verificar locks bloqueantes e espaço em disco. Se disco > 90%, escalar para DBA.

4

Se não resolveu: Escalar

Contactar DBA de plantão via PagerDuty. Incluir output dos passos anteriores na mensagem.

💡 Dica Prática

Crie um runbook para cada alerta que existe. Se um alerta dispara e não tem runbook, ou o alerta é inútil (delete) ou falta documentação (crie). O objetivo: qualquer pessoa do time deve conseguir seguir o runbook sem ajuda, mesmo às 3h da manhã.

🔄 Chaos Engineering

Quebre em controlled para não quebrar no real. Chaos Engineering é a prática de introduzir falhas deliberadas em produção para testar a resiliência do sistema: matar instâncias, injetar latência, simular network partition. Você descobre fraquezas antes dos usuários.

💡 Princípios do Chaos Engineering

  • •Steady State Hypothesis — Defina o comportamento normal antes de injetar falha. “Latência p99 < 200ms”
  • •Blast Radius Control — Limite o impacto. Começe com 1% do tráfego, não 100%
  • •Game Days — Exercícios agendados onde o time simula incidentes e pratica resposta
  • •Automação — Chaos Monkey (Netflix), Gremlin, Litmus (Kubernetes) automatizam injeção de falhas

📊 Caso Real: Netflix Chaos Monkey

A Netflix inventou Chaos Engineering e roda em produção desde 2011:

  • Chaos Monkey: Mata instâncias EC2 aleatórias em produção durante horário comercial
  • Chaos Kong: Simula a queda de uma região inteira da AWS (US-East-1)
  • Resultado: Quando a AWS US-East-1 realmente caiu em 2017, Netflix continuou funcionando enquanto metade da internet caiu
  • Filosofia: “Se não testamos em produção, não sabemos se funciona em produção”

✓ O que FAZER

  • ✓Começar com Game Days antes de automatizar
  • ✓Ter rollback instantâneo para cada experimento
  • ✓Rodar chaos em horário comercial (quando o time está presente)

✗ O que NÃO fazer

  • ✗Rodar chaos em produção sem observabilidade adequada
  • ✗Começar com 100% do tráfego no primeiro teste
  • ✗Fazer chaos sem avisar o time (isso é sabotagem, não engenharia)

📊 SRE — Site Reliability Engineering

Quando ops vira engenharia. SRE é a disciplina criada pelo Google que aplica engenharia de software a problemas de operações. O foco é automação, SLOs, error budgets e redução de toil. Em vez de escalar o time linearly com a infra, SRE escala com automação.

💡 Os Pilares do SRE

  • •SLOs (Service Level Objectives) — Meta interna de confiabilidade. Ex: 99.9% de uptime, p99 < 200ms
  • •Error Budgets — O inverso do SLO. Se SLO é 99.9%, você tem 0.1% de “budget” para errar. Enquanto tem budget: ship features. Budget esgotou: pare e melhore confiabilidade
  • •Toil — Trabalho manual, repetitivo, automatizável. SRE busca manter toil < 50% do tempo. O resto é engenharia
  • •Automação sobre processo — Se você precisa de um checklist, precisa de um script. Se precisa de um script, precisa de um serviço

📊 Como o Error Budget Funciona na Prática

O Google definiu a regra que alinha product e SRE automaticamente:

  • SLO: 99.9% → Error budget: 43 minutos de downtime por mês
  • Budget OK: Time de produto prioriza features. Deploy quando quiser
  • Budget baixo: Congela features. Prioriza confiabilidade, testes, rollback automático
  • Budget estourado: Freeze total. Só fixes de confiabilidade até o budget regenerar
  • Resultado: Não precisa de discussão política — o número decide

💡 Dica Prática

Comece definindo SLOs para seus 3 endpoints mais críticos. Não precisa de um time de SRE dedicado — a mentalidade SRE pode ser adotada por qualquer dev. Defina: “Login deve ter p99 < 300ms e 99.95% de success rate”. Monitore. Quando o error budget ficar baixo, priorize estabilidade.

🔄 On-Call

Plantão que não destrói sua vida. On-call é a rotação de pessoas responsáveis por responder a incidentes fora do horário. Mal feito, causa burnout e turnover. Bem feito, é sustentável e até recompensador. A diferença está em: rotação justa, qualidade de alertas, compensação e ferramentas adequadas.

💡 Boas Práticas de On-Call

  • •Rotação semanal — Mínimo de 4-5 pessoas na rotação para ser sustentável
  • •Máximo 1-2 incidentes por noite — Se mais, os alertas estão errados, não a pessoa
  • •Compensação — Folga compensatória ou adicional financeiro. On-call não é voluntário
  • •Secondary on-call — Backup que é acionado se o primário não responde em 5 minutos
  • •Runbook para cada alerta — Sem runbook = alerta que não deveria existir

⚠️ Sinais de On-Call Tóxico

Se você reconhece esses padrões, o problema é a engenharia, não as pessoas:

  • •On-call é acordado toda noite — alertas estão errados ou o sistema é frágil demais
  • •As mesmas pessoas ficam sempre de plantão porque “só elas sabem”
  • •Alertas disparam e o on-call não tem ação clara a tomar
  • •Não existe compensação por ficar de plantão

💡 Dica Prática

Faça um “alert review” mensal: liste todos os alertas que dispararam, classifique como “actionable” ou “noise”. Delete os noise. Para cada actionable, garanta que existe runbook. Meta: cada alerta que acorda alguém deve ter ação imediata necessária. Se não tem, é noise.

📈 Reliability Patterns

Sem esses padrões, uma falha cascateia pelo sistema inteiro. Circuit breaker, retry com backoff, bulkhead e timeout são os 4 padrões fundamentais de resiliência. Eles transformam falhas catastróficas em degradações controladas.

💡 Os 4 Padrões de Resiliência

  • •Circuit Breaker — Para de chamar serviço que está falhando. 3 estados: closed (normal), open (bloqueado), half-open (testando). Evita cascading failure
  • •Retry com Exponential Backoff — Tenta de novo, mas esperando mais a cada tentativa: 100ms, 200ms, 400ms + jitter. Sem jitter, milhares de clients retentam juntos
  • •Bulkhead — Isola recursos por domínio. Pool de conexões separado por serviço. Se um esgota, os outros continuam
  • •Timeout — Nunca espere para sempre. Defina timeout para toda chamada externa. Budget de timeout: se total é 2s, cada hop tem no máximo 500ms

⚠️ Sem Reliability Patterns: Cascading Failure

Anatomia de uma falha em cascata sem circuit breaker:

  • 1. Serviço de pagamento fica lento (banco sobrecarregado)
  • 2. Serviço de pedido espera resposta → timeout → pool de conexões esgota
  • 3. Serviço de pedido para de responder → serviço de catálogo também espera
  • 4. Catálogo cai → frontend sem dados → TUDO fora do ar
  • Com circuit breaker: Pagamento cai → circuit abre → pedido retorna “tente mais tarde” → catálogo e frontend continuam funcionando normalmente

💡 Dica Prática

Implemente timeout + circuit breaker em toda chamada HTTP entre serviços. Em Node.js, use opossum para circuit breaker. Em Java, Resilience4j. Em Go, sony/gobreaker. Nunca deixe uma chamada externa sem timeout — o default “infinito” é a maior armadilha de sistemas distribuídos.
