📄 O que é IaC

Infrastructure as Code (IaC) é a prática de gerenciar e provisionar infraestrutura através de arquivos de configuração em vez de processos manuais. Assim como código de aplicação, a infraestrutura pode ser versionada, revisada, testada e auditada. Se você configura servidores clicando em interfaces web, você está criando snowflakes — servidores únicos e irreproduíveis.

💡 Imperativo vs. Declarativo

Duas abordagens fundamentais para definir infraestrutura:

  • •Imperativo — Você descreve os passos: “crie o servidor, instale o nginx, configure a porta 80” (Ansible, scripts bash)
  • •Declarativo — Você descreve o estado desejado: “quero um servidor com nginx na porta 80” e a ferramenta resolve como chegar lá (Terraform, Kubernetes)
  • •Idempotência — Rodar o mesmo código múltiplas vezes deve produzir o mesmo resultado. Essencial para confiabilidade

📊 Caso Real: Capital One

A Capital One migrou toda sua infraestrutura para IaC e documentou os resultados:

  • 80% de redução no tempo de provisionamento de novos ambientes
  • Zero configuração manual — Todo servidor é criado por código e pode ser reconstruído em minutos
  • Audit trail completo — Cada mudança de infraestrutura tem PR, review e histórico no Git
  • Conformidade regulatória — PCI-DSS compliance verificado automaticamente via políticas como código

✓ O que FAZER

  • ✓Tratar infraestrutura como cattle, não pets — descartável e recriável
  • ✓Versionar toda configuração no mesmo repositório ou monorepo
  • ✓Fazer code review em mudanças de infraestrutura como em código

✗ O que NÃO fazer

  • ✗Fazer mudanças manuais em produção “só pra testar”
  • ✗Manter snowflake servers que ninguém sabe como foram configurados
  • ✗Ignorar drift detection — quando o estado real diverge do código

💡 Dica Prática

Comece com o “teste do fogo”: consiga destruir e recriar toda a infraestrutura de um ambiente a partir do código, sem nenhuma intervenção manual. Se não conseguir, você ainda não tem IaC de verdade — tem scripts de automação parcial.

⚡ Terraform

Terraform é a ferramenta de IaC mais popular do mundo, criada pela HashiCorp. Usa HCL (HashiCorp Configuration Language) para declarar recursos em qualquer cloud provider: AWS, GCP, Azure, Cloudflare e centenas de outros. O conceito de “plan antes de apply” dá visibilidade total sobre o que vai mudar antes de executar.

💡 Ciclo de Vida do Terraform

  • •terraform init — Inicializa o projeto, baixa providers e configura o backend de estado
  • •terraform plan — Mostra o que vai ser criado, modificado ou destruído. Sem executar nada
  • •terraform apply — Executa as mudanças planejadas na infraestrutura real
  • •terraform destroy — Remove todos os recursos gerenciados. Útil para ambientes temporários
  • •State file — Arquivo que mapeia recursos declarados a recursos reais. Deve ser armazenado remotamente (S3, GCS)

⚠️ Armadilhas do Terraform

  • State file local — Nunca comite o state file no Git. Use remote state com locking (S3 + DynamoDB)
  • Blast radius — Um único terraform apply pode destruir infraestrutura inteira. Use workspaces e módulos para isolar
  • Drift — Alguém mudou algo manualmente no console? O state divergiu. Use terraform plan regularmente para detectar

💡 Dica Prática

Estruture seus projetos Terraform com módulos reutilizáveis. Crie módulos para padrões comuns (VPC, banco de dados, cluster ECS) e reutilize entre ambientes e projetos. Isso evita duplicação e garante consistência. Use Terragrunt para gerenciar múltiplos ambientes com DRY.

📦 Docker Compose

Docker Compose é a ferramenta para definir e executar aplicações multi-container. Um único arquivo docker-compose.yml descreve todos os serviços, redes e volumes. Ideal para ambiente de desenvolvimento local e testes de integração, onde você precisa de banco, cache, fila de mensagens e sua aplicação rodando juntos.

💡 Componentes do Compose

  • •Services — Cada container é um serviço: app, db, redis, rabbitmq
  • •Networks — Redes isoladas para comunicação entre containers. DNS automático por nome do serviço
  • •Volumes — Persistência de dados entre restarts. Essencial para bancos de dados locais
  • •Profiles — Agrupe serviços por cenário: --profile debug sobe ferramentas extras
  • •Depends_on — Ordem de inicialização e health checks para garantir que dependências estão prontas

✓ O que FAZER

  • ✓Usar .env files para configurações específicas do desenvolvedor
  • ✓Configurar health checks em depends_on para esperar serviços ficarem prontos
  • ✓Documentar o setup local com um único comando: docker compose up

✗ O que NÃO fazer

  • ✗Usar Docker Compose em produção para workloads críticos
  • ✗Hardcodar senhas e tokens no docker-compose.yml
  • ✗Ignorar limites de memória e CPU nos serviços locais

💡 Dica Prática

Crie um Makefile ou Taskfile com targets como make dev, make test, make seed que abstraem os comandos do Docker Compose. Um novo desenvolvedor deve conseguir rodar o projeto inteiro em menos de 5 minutos.

☸️ Kubernetes

Kubernetes (K8s) é o orquestrador de containers padrão da indústria. Gerencia deploy, escalonamento e operação de aplicações containerizadas em clusters de máquinas. Criado pelo Google com base em 15 anos de experiência rodando Borg. Poderoso, mas com complexidade significativa — não é para todo projeto.

💡 Objetos Fundamentais do K8s

  • •Pod — Menor unidade deployável. Um ou mais containers que compartilham rede e storage
  • •Deployment — Gerencia réplicas de pods, rolling updates e rollbacks
  • •Service — Endpoint estável para acessar pods. Load balancing automático
  • •Ingress — Roteamento HTTP/HTTPS externo para services internos
  • •ConfigMap / Secret — Configurações e credenciais injetadas nos pods
  • •HPA — Horizontal Pod Autoscaler: escala automaticamente baseado em CPU, memória ou métricas custom

📊 Caso Real: Spotify

A migração do Spotify para Kubernetes é um exemplo de escala massiva:

  • 2.500+ serviços rodando em Kubernetes com GKE
  • Backstage — Criaram a plataforma open-source de developer portal para abstrair a complexidade do K8s
  • 10x mais rápido para provisionar novos serviços vs. infraestrutura manual anterior
  • Lição: K8s sozinho não resolve — você precisa de uma plataforma acima dele

💡 Dica Prática

Antes de adotar Kubernetes, pergunte: “Tenho mais de 10 serviços para orquestrar?” Se não, considere alternativas mais simples: ECS, Cloud Run, Railway, Fly.io. K8s é poderoso, mas a complexidade operacional é real. Use managed K8s (EKS, GKE, AKS) para reduzir a carga operacional do control plane.

🔐 Secrets Management

Credenciais vazadas são a causa número 1 de brechas de segurança. Gerenciar secrets — senhas de banco, chaves de API, tokens de serviço — de forma segura e auditável é crítico. Nunca no código, nunca em plain text, sempre com rotação automática.

💡 Ferramentas e Padrões

  • •HashiCorp Vault — Padrão da indústria. Secrets dinâmicos, rotação automática, audit log completo
  • •AWS Secrets Manager / Parameter Store — Soluções nativas AWS com integração a IAM
  • •SOPS + Age — Encriptação de arquivos de secrets que podem ser commitados no Git com segurança
  • •Sealed Secrets (K8s) — Secrets encriptados que só o cluster pode decodificar

⚠️ Caso Real: Uber (2016)

Um dos maiores vazamentos de dados da história começou com secrets mal gerenciados:

  • Credenciais AWS commitadas em um repositório privado do GitHub
  • 57 milhões de dados de usuários e motoristas expostos
  • US$148 milhões em multas e acordos judiciais
  • Lição: secrets em código é uma bomba-relógio, mesmo em repos privados

💡 Dica Prática

Configure git-secrets ou gitleaks como pre-commit hook. Eles impedem que credenciais sejam commitadas acidentalmente. Além disso, rode scans regulares no histórico do Git — secrets removidos do código atual ainda existem em commits antigos.

🌐 CDN e Edge

CDNs (Content Delivery Networks) e edge computing aproximam o conteúdo e a computação do usuário. Em vez de toda requisição viajar até um servidor central, o conteúdo é servido de pontos de presença (PoPs) distribuídos globalmente. Isso reduz latência, melhora performance e aumenta a resiliência.

💡 CDN vs. Edge Computing

  • •CDN Tradicional — Cache de conteúdo estático (imagens, JS, CSS). Cloudflare, CloudFront, Fastly
  • •Edge Functions — Execução de código na borda da rede. Cloudflare Workers, Vercel Edge Functions, Deno Deploy
  • •Edge Databases — Dados replicados globalmente. Turso (LibSQL), PlanetScale, Neon
  • •Cache Invalidation — O problema mais difícil da computação. TTL, purge manual, stale-while-revalidate

📊 Impacto na Performance

  • 40-60% de redução na latência para usuários distantes do servidor de origem
  • 90%+ do tráfego pode ser servido pelo cache da CDN para sites com conteúdo estático
  • Cloudflare Workers — Cold start de <5ms vs. 100-500ms de Lambda/Cloud Functions
  • Cada 100ms de latência custam 1% de conversão em e-commerce (dados Amazon)

💡 Dica Prática

Comece com CDN para assets estáticos (imagens, JS, CSS) — é o quick win mais fácil e de maior impacto. Depois, explore stale-while-revalidate para APIs que podem tolerar dados ligeiramente desatualizados. Edge Functions ficam para quando você precisa de personalização por região ou A/B testing com latência mínima.

💰 Otimização de Custos

Cloud sem controle de custos é um cartão de crédito sem limite em mãos erradas. FinOps (Financial Operations) é a disciplina de otimizar custos de infraestrutura cloud sem sacrificar performance. Recursos ociosos, over-provisioning e falta de visibilidade são os maiores desperdiciais.

💡 Estratégias de Otimização

  • •Right-sizing — Ajustar recursos ao uso real. A maioria dos servidores usa <20% da capacidade provisionada
  • •Reserved Instances / Savings Plans — Compromisso de 1-3 anos para 40-70% de desconto
  • •Spot Instances — Capacidade excedente com 60-90% de desconto. Ideal para workloads toleráveis a interrupção
  • •Auto-scaling — Escalar para cima quando necessário e, crucialmente, escalar para baixo quando não
  • •Tagging — Tag em todo recurso para saber qual time/projeto está gastando o quê

📊 Caso Real: 37signals (Basecamp/HEY)

Em 2023, a 37signals publicou os números de sua migração da cloud para servidores próprios:

  • US$3.2M/ano — Custo anterior na AWS
  • US$840K/ano — Custo com servidores próprios (colocation + hardware)
  • US$7M economizados em 5 anos (projetado)
  • Contexto: Carga previsível, time experiente, sem necessidade de escala elástica

Cuidado: Isso não significa que todo mundo deve sair da cloud. A 37signals tem contexto específico que viabiliza.

💡 Dica Prática

Implemente alertas de custo desde o dia 1. Configure budgets no AWS/GCP/Azure que notificam quando os gastos ultrapassam thresholds (50%, 80%, 100% do budget). Além disso, agende a destruição automática de ambientes de staging fora do horário comercial — isso sozinho pode reduzir 30-40% da fatura.
