🛡️ OWASP Top 10

O OWASP Top 10 é a lista atualizada dos 10 riscos mais críticos em aplicações web. Injection, Broken Authentication, Sensitive Data Exposure, XSS — esses 10 itens causam aproximadamente 90% dos breaches. Conhecê-los é o mínimo obrigatório para qualquer desenvolvedor que coloca código em produção.

💡 Os 10 Riscos Mais Críticos (2021+)

Cada um destes já causou prejuízos de bilhões de dólares:

  • •A01: Broken Access Control — Usuários acessam recursos que não deveriam (subiu para #1 em 2021)
  • •A02: Cryptographic Failures — Dados sensíveis sem encriptação adequada
  • •A03: Injection — SQL, NoSQL, OS, LDAP injection via input não sanitizado
  • •A04: Insecure Design — Falhas de design que nenhum código corrige (novo em 2021)
  • •A05: Security Misconfiguration — Defaults inseguros, headers faltando, permissões excessivas

📊 Caso Real: Equifax (2017)

Um dos maiores breaches da história — dados de 147 milhões de pessoas expostos:

  • Causa: Vulnerabilidade conhecida no Apache Struts (CVE-2017-5638) — patch disponível há 2 meses
  • OWASP: A06 (Vulnerable Components) + A05 (Security Misconfiguration)
  • Custo: US$1.4 bilhão em multas e remediação
  • Lição: Um simples npm audit fix ou Dependabot teria prevenido tudo

✓ O que FAZER

  • ✓Usar prepared statements para toda query de banco
  • ✓Validar e sanitizar todo input do usuário (server-side)
  • ✓Implementar rate limiting em endpoints de autenticação

✗ O que NÃO fazer

  • ✗Confiar em validação client-side como única defesa
  • ✗Concatenar input do usuário diretamente em queries SQL
  • ✗Deixar stack traces visíveis em respostas de produção

💡 Dica Prática

Imprima o OWASP Top 10 e cole perto do seu monitor. Antes de cada PR, passe pelo checklist mental: “Esse código aceita input externo? Está sanitizado? Os acessos estão autorizados? Dados sensíveis estão encriptados?” Em 30 segundos você previne 90% das vulnerabilidades.

🔑 Autenticação Moderna

Passwords são legado. A evolução da autenticação vai de passwords → MFA → passwordless (magic links, passkeys, biometrics). OAuth2 e OpenID Connect são os protocolos padrão para social login e SSO. Entender essa evolução é crítico porque 80% dos breaches envolvem credenciais comprometidas.

💡 A Evolução da Autenticação

De passwords para passwordless — cada geração resolve as fraquezas da anterior:

  • •Passwords — Reutilizadas, fracas, phisháveis. Ainda dominam, mas estão morrendo
  • •MFA (Multi-Factor Auth) — Algo que você sabe + algo que você tem. Reduz risco em 99.9% (Microsoft)
  • •OAuth2 / OIDC — Delega autenticação para Google, GitHub, etc. Você não armazena senhas
  • •Passkeys (WebAuthn) — Chave criptográfica bound ao domínio. Imune a phishing. Apple, Google e Microsoft suportam nativamente
  • •Magic Links — Email com link temporário. Sem senha para lembrar. Slack, Notion usam

📊 Dados: Verizon DBIR 2024

  • 80% dos breaches envolvem credenciais comprometidas ou fracas
  • MFA bloqueia 99.9% dos ataques de credential stuffing (Microsoft Security)
  • Passkeys eliminam phishing completamente — a autenticação é vinculada ao domínio
  • Supabase Auth / Auth0 / Clerk implementam tudo isso em minutos, não meses

💡 Dica Prática

Nunca implemente auth do zero. Use Supabase Auth, Auth0, Clerk ou Firebase Auth. Cada linha de código de autenticação que você escreve é uma linha onde pode errar. Se precisar de JWT, use jose (lib) — nunca parse JWT manualmente. E sempre valide a assinatura no server, nunca no client.

🔒 HTTPS e TLS

Encriptação não é opcional. TLS encripta a comunicação entre client e server. HTTPS = HTTP + TLS. Browsers marcam sites HTTP como “Not Secure”, Google penaliza no SEO, e dados em trânsito sem TLS podem ser interceptados por qualquer pessoa na mesma rede.

💡 Camadas de Encriptação

  • •TLS 1.3 — Versão mais recente, handshake mais rápido (1-RTT vs 2-RTT), mais seguro
  • •Let’s Encrypt — Certificados gratuitos, renovação automática a cada 90 dias via certbot
  • •Cloudflare SSL — TLS automático para qualquer domínio. Zero config
  • •HSTS — Header que força HTTPS. Sem ele, o primeiro request pode ser HTTP
  • •mTLS — Mutual TLS para comunicação service-to-service. Ambos os lados se autenticam

✓ O que FAZER

  • ✓Usar HSTS com max-age=31536000; includeSubDomains
  • ✓Configurar redirect 301 de HTTP para HTTPS
  • ✓Usar mTLS entre serviços internos em produção

✗ O que NÃO fazer

  • ✗Usar TLS 1.0 ou 1.1 — são vulneráveis e deprecated
  • ✗Ignorar certificate warnings no código com rejectUnauthorized: false
  • ✗Deixar certificados expirarem por falta de automação

💡 Dica Prática

Teste seu site em ssllabs.com/ssltest — ele analisa a configuração TLS e dá nota de A a F. Busque nota A+. E adicione o header Strict-Transport-Security para garantir que browsers sempre usem HTTPS após a primeira visita.

🏰 Zero Trust

“Nunca confie, sempre verifique.” O modelo Zero Trust assume que nada é confiável por padrão — nem a rede interna, nem os serviços próprios. Cada request é autenticada, autorizada e encriptada, independente de onde vem. O modelo de “perímetro seguro” (firewall) falha quando o atacante já está dentro.

💡 Princípios do Zero Trust

  • •Identity-based access — Acesso baseado em identidade verificada, não em localização de rede
  • •Micro-segmentation — Cada serviço só se comunica com quem precisa, nada mais
  • •Least privilege — Mínimo de permissões necessárias, por menor tempo possível
  • •Continuous verification — Não basta autenticar uma vez — verifique continuamente

⚠️ Caso Real: SolarWinds (2020)

Um dos ataques mais sofisticados da história — e Zero Trust teria limitado drasticamente o impacto:

  • O ataque: Atacantes comprometeram o build system do SolarWinds e injetaram malware no update
  • Resultado: 18.000 organizações instalaram o update infectado, incluindo agências do governo dos EUA
  • 9 meses sem detecção: A rede interna era “trusted”, então lateral movement não disparou alertas
  • Com Zero Trust: Cada lateral movement exigiria autenticação, e a comunicação anômala seria detectada

💡 Dica Prática

Comece implementando Zero Trust de forma incremental: 1) mTLS entre serviços, 2) Identity-aware proxy (como o BeyondCorp do Google), 3) Políticas de least privilege em IAM. Não precisa migrar tudo de uma vez — cada camada adiciona segurança.

📋 LGPD na Prática

A Lei Geral de Proteção de Dados não é só um problema jurídico — é um problema de arquitetura. Consentimento, minimização de dados, direito de exclusão, portabilidade e notificação de breach exigem que o sistema seja desenhado para compliance desde o início. Multas podem chegar a 2% do faturamento.

💡 O que a LGPD Exige do Desenvolvedor

  • •Consentimento — Coletar dados pessoais só com consentimento explícito e específico
  • •Minimização — Coletar apenas os dados estritamente necessários para o propósito
  • •Right to be forgotten — DELETE de todos os dados do usuário em todas as tabelas, caches, backups e terceiros
  • •Portabilidade — Exportar todos os dados pessoais em formato legível
  • •Notificação de breach — Comunicar a ANPD e usuários afetados em prazo razoável
  • •Audit trail — Registrar quem acessou dados pessoais, quando e por quê

✓ O que FAZER

  • ✓Ter uma tabela/serviço de consent management
  • ✓Encriptar dados pessoais at rest e in transit
  • ✓Implementar endpoint de data export para portabilidade

✗ O que NÃO fazer

  • ✗Logar dados pessoais (CPF, email) em plain text nos logs
  • ✗Espalhar dados pessoais por dezenas de tabelas sem mapeamento
  • ✗Usar soft delete sem TTL — dados “deletados” que ficam para sempre

💡 Dica Prática

Crie um Data Map: um documento que lista onde cada tipo de dado pessoal é armazenado (tabela, cache, log, backup, serviço externo). Sem esse mapa, implementar “right to be forgotten” é impossível. Ferramentas como OneTrust e Securiti ajudam a automatizar.

🔍 Dependency Scanning

Suas dependências são seu risco. Aplicações modernas têm centenas de dependências transitivas. Cada uma é um vetor de ataque potencial. CVEs em libs desatualizadas são o vetor mais comum de ataque em 2024. Se você não monitora, não sabe se está vulnerável.

💡 O Ecossistema de Scanning

  • •Dependabot (GitHub) — Abre PR automático quando encontra CVE nas suas deps. Merge = seguro
  • •Snyk — Scanning profundo com fix automático. Suporta containers e IaC também
  • •npm audit / pip-audit — Ferramentas nativas do ecossistema. npm audit fix resolve automaticamente
  • •SBOM — Software Bill of Materials: inventário de todas as dependências. Exigido por regulamentações

⚠️ Caso Real: Log4Shell (2021)

A vulnerabilidade mais impactante da década — e um alerta sobre dependências:

  • CVE-2021-44228: Vulnerabilidade crítica no Log4j (lib de logging Java)
  • Impacto: 35.000+ pacotes Java afetados, incluindo Minecraft, iCloud, Twitter, Steam
  • Severidade: CVSS 10.0 (máximo) — execução remota de código com uma string de log
  • Lição: Quem tinha Dependabot/Snyk foi alertado em horas. Quem não tinha, levou semanas para descobrir se era afetado

💡 Dica Prática

Ative o Dependabot agora — leva 2 minutos. Vá em Settings → Security → Enable Dependabot no seu repositório GitHub. Adicione também npm audit no seu pipeline de CI como step obrigatório. Se falhar, o deploy não acontece.

🧪 Security Testing

Bugs de segurança pegos em code review custam 10x menos que em produção, e 100x menos que após um breach. SAST, DAST e pentest são as três camadas de security testing que todo sistema sério precisa. Automatize SAST e DAST na CI — pentest periodicamente.

💡 As 3 Camadas de Security Testing

  • •SAST (Static Analysis) — Analisa o código fonte sem executar. Semgrep, CodeQL. Roda em cada PR. Encontra SQL injection, XSS, path traversal automaticamente
  • •DAST (Dynamic Analysis) — Testa a aplicação rodando. OWASP ZAP, Burp Suite. Simula ataques reais contra endpoints
  • •Pentest — Hacker ético atacando o sistema com criatividade humana. Encontra falhas lógicas que ferramentas não pegam
  • •Bug Bounty — Programa público onde pesquisadores reportam vulnerabilidades por recompensa. HackerOne, Bugcrowd

🛠️ Pipeline de Segurança Automatizado

1

Pre-commit: Secrets Scanning

git-secrets ou truffleHog scaneia cada commit por API keys, passwords e tokens hardcoded. Bloqueia antes do push.

2

CI: SAST + Dependency Scan

Semgrep/CodeQL analisa código, Dependabot/Snyk verifica CVEs. Roda em cada PR. Se encontrar vuln crítica, bloqueia merge.

3

Staging: DAST

OWASP ZAP roda contra staging após deploy. Testa endpoints reais por injection, XSS, auth bypass.

4

Trimestral: Pentest

Hacker ético com escopo definido. Encontra falhas de lógica de negócio, privilege escalation, e vulnerabilidades criativas que ferramentas não detectam.

💡 Dica Prática

Comece pelo mais fácil: ative o GitHub Code Scanning (CodeQL) — é gratuito para repos públicos e roda SAST em cada PR automaticamente. Em 5 minutos de configuração, você tem um security scanner que encontra SQL injection, XSS e path traversal sem esforço manual.
