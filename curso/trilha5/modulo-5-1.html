🔄 O que é CI/CD

CI/CD é a espinha dorsal da engenharia moderna. Continuous Integration (CI) garante que cada alteração de código seja integrada, compilada e testada automaticamente. Continuous Delivery (CD) garante que o código esteja sempre pronto para ir a produção. Continuous Deployment vai além: cada commit aprovado vai direto para produção, sem intervenção humana.

💡 Os 3 Estágios do CI/CD

Cada estágio aumenta a maturidade e a confiança no processo:

  • •Continuous Integration — Merge frequente no branch principal, build automático, testes unitários rodando a cada push
  • •Continuous Delivery — Artefato pronto para deploy a qualquer momento, mas alguém aperta o botão manualmente
  • •Continuous Deployment — Cada commit que passa nos testes vai automaticamente para produção. Exige maturidade máxima

📊 Dados: DORA State of DevOps

O relatório DORA (Google) analisa milhares de times e comprova o impacto do CI/CD:

  • Elite performers — Deploy múltiplas vezes por dia, lead time < 1 hora
  • Low performers — Deploy mensal ou trimestral, lead time de 1-6 meses
  • 208x mais deploys e 106x menor lead time nos times elite vs. low performers
  • 7x menor taxa de falha em mudanças e 2.604x mais rápido na recuperação

✓ O que FAZER

  • ✓Fazer merge no branch principal pelo menos 1x por dia
  • ✓Rodar testes automatizados a cada push, sem exceções
  • ✓Tratar o pipeline quebrado como prioridade máxima

✗ O que NÃO fazer

  • ✗Manter branches de longa duração sem merge por semanas
  • ✗Ter testes que falham intermitentemente (flaky tests) sem consertar
  • ✗Fazer deploy manual “só dessa vez” porque é urgente

💡 Dica Prática

Comece com o mínimo viável: lint + testes unitários + build rodando no push. Depois adicione testes de integração, security scanning e deploy automatizado. Um pipeline de 10 minutos que roda sempre é infinitamente melhor que um pipeline de 2 horas que ninguém espera terminar.

⚙️ GitHub Actions

GitHub Actions é a plataforma de CI/CD mais popular do mundo, integrada diretamente ao repositório. Workflows declarativos em YAML definem pipelines que respondem a eventos como push, pull request ou schedule. A integração nativa com o ecossistema GitHub e o marketplace de actions torna a configuração rápida e poderosa.

💡 Anatomia de um Workflow

Os componentes fundamentais de um pipeline GitHub Actions:

  • •Trigger (on) — Evento que inicia o workflow: push, pull_request, schedule, workflow_dispatch
  • •Jobs — Unidades de trabalho que rodam em paralelo ou sequencialmente em runners
  • •Steps — Comandos individuais dentro de um job (run ou uses)
  • •Actions — Componentes reutilizáveis do marketplace (checkout, setup-node, cache)
  • •Secrets — Variáveis criptografadas para tokens, senhas e chaves de API

📊 Caso Real: Vercel

A Vercel usa GitHub Actions como base de todo o pipeline open-source do Next.js:

  • 500+ PRs por semana — Cada uma passa por CI automático com testes, lint e build
  • Matrix builds — Testes rodam em múltiplas versões do Node.js e sistemas operacionais simultaneamente
  • Canary releases — Versões de teste são publicadas automaticamente no npm a cada merge

✓ O que FAZER

  • ✓Usar cache de dependências para acelerar o pipeline
  • ✓Versionar actions com hash do commit, não tags mutáveis
  • ✓Separar jobs de test, build e deploy para paralelismo

✗ O que NÃO fazer

  • ✗Colocar secrets hardcoded no YAML do workflow
  • ✗Usar actions de terceiros sem auditar o código-fonte
  • ✗Criar workflows com +30 minutos de execução sem otimizar

💡 Dica Prática

Use act para rodar GitHub Actions localmente antes de commitar. Isso evita o ciclo frustrante de push → esperar pipeline → falhou → corrigir → push de novo. Teste localmente, commite com confiança.

📦 Docker

Docker revolucionou a forma como empacotamos e distribuímos software. Containers garantem que o ambiente de desenvolvimento seja idêntico ao de produção, eliminando o clássico “na minha máquina funciona”. Imagens imutáveis, leves e portáveis são a base de toda pipeline moderna de CI/CD.

💡 Conceitos Essenciais do Docker

Os blocos fundamentais para trabalhar com containers:

  • •Dockerfile — Receita declarativa para construir uma imagem: FROM, COPY, RUN, CMD
  • •Image — Snapshot imutável com tudo que a aplicação precisa para rodar
  • •Container — Instância em execução de uma imagem, isolada do host
  • •Registry — Repositório de imagens (Docker Hub, GitHub Container Registry, ECR)
  • •Multi-stage build — Múltiplos estágios no Dockerfile para imagens finais menores e mais seguras

⚠️ Erros Comuns com Docker

Erros que custam horas de debugging e comprometem segurança:

  • Rodar como root — 58% das imagens Docker Hub rodam como root, criando vulnerabilidades críticas
  • Imagens gigantes — Usar ubuntu:latest (77MB) quando alpine (5MB) resolve. Imagens menores = deploys mais rápidos
  • Secrets no build — Colocar chaves de API no Dockerfile expõe credenciais em cada layer da imagem
  • Cache invalidado — Copiar todo o código antes de instalar dependências invalida o cache desnecessariamente

💡 Dica Prática

Sempre use multi-stage builds. O primeiro estágio compila e testa, o segundo copia apenas o artefato final. Resultado: imagens de produção 10x menores, sem compiladores, ferramentas de debug ou código-fonte expostos.

🎯 Deploy Strategies

A estratégia de deploy define como uma nova versão substitui a anterior em produção. Não existe estratégia universalmente melhor — cada uma equilibra velocidade, risco e complexidade de maneiras diferentes. A escolha certa depende do seu SLA, da tolerância a downtime e da maturidade do pipeline.

💡 As 5 Estratégias Principais

  • •Rolling Update — Substitui instâncias gradualmente. Zero downtime, mas duas versões coexistem temporariamente
  • •Blue/Green — Dois ambientes idênticos. Troca de tráfego instantânea. Rollback em segundos, mas custo dobrado
  • •Canary — Nova versão para 1-5% do tráfego primeiro. Validar métricas antes de expandir para 100%
  • •Feature Flags — Deploy do código sem ativar a feature. Liga/desliga funcionalidades por usuário, região ou percentual
  • •Recreate — Derruba tudo, sobe a nova versão. Simples mas com downtime. Aceitável apenas em ambientes internos

📊 Caso Real: Google e Canary Deploys

O Google faz centenas de deploys por dia usando canary releases em vários estágios:

  • 0.1% — Tráfego inicial para validar métricas básicas (erros, latência)
  • 1% — Validação com mais volume por 30 minutos
  • 10% — Acompanhamento de métricas de negócio por 1 hora
  • 100% — Rollout completo apenas se todos os gates passaram

Rollback automático acontece se qualquer métrica degradar além do threshold definido.

💡 Dica Prática

Para a maioria das aplicações, comece com Rolling Update + Feature Flags. É a combinação mais versátil: deploy contínuo sem downtime e controle granular sobre o que cada usuário vê. Blue/Green e Canary ficam para quando você precisar de garantias adicionais de segurança.

🌐 Ambientes

Ambientes são cópias controladas do sistema em diferentes estágios de maturidade. Cada ambiente serve um propósito específico no ciclo de vida do software. A gestão correta de ambientes é o que separa times que fazem deploy com confiança de times que rezam antes de cada release.

💡 A Escada de Ambientes

  • •Local (dev) — Máquina do desenvolvedor. Docker Compose para subir dependências
  • •CI — Ambiente efêmero criado automaticamente pelo pipeline para rodar testes
  • •Staging — Réplica de produção para validação final. Mesma infra, dados anonimizados
  • •Production — O ambiente real. Usuários reais, dados reais, consequências reais

✓ O que FAZER

  • ✓Manter staging o mais parecido possível com produção
  • ✓Usar variáveis de ambiente para diferenciar configurações
  • ✓Automatizar a criação e destruição de ambientes efêmeros

✗ O que NÃO fazer

  • ✗Testar em produção sem feature flags ou canary
  • ✗Usar dados reais de produção em staging sem anonimizar
  • ✗Manter staging permanentemente divergente de produção

💡 Dica Prática

Considere preview environments (Vercel, Netlify, Railway): cada PR cria automaticamente um ambiente temporário com a versão do código daquela branch. Isso permite QA visual, testes de integração e revisão por stakeholders antes do merge.

🔃 Rollback

O deploy mais seguro é aquele que pode ser desfeito em segundos. Rollback é a capacidade de reverter para a versão anterior quando algo dá errado. Sem uma estratégia clara de rollback, cada deploy é uma aposta — e apostar em produção é a receita para incidentes.

💡 Tipos de Rollback

  • •Rollback de imagem — Volta para a imagem Docker anterior. Rápido e confiável
  • •Rollback de configuração — Reverte variáveis de ambiente ou feature flags
  • •Rollback de banco — O mais perigoso. Migrações destrutivas são irreversíveis. Use migrações expand/contract
  • •Rollback de tráfego — Redireciona tráfego para a versão anterior (Blue/Green)

📊 Caso Real: GitLab (2017)

Em janeiro de 2017, o GitLab sofreu um dos incidentes mais famosos da história do DevOps:

  • Um engenheiro deletou acidentalmente o banco de dados de produção
  • 5 estratégias de backup — nenhuma funcionava. Backups nunca foram testados
  • 300GB de dados perdidos — recuperação parcial levou 18 horas
  • Lição definitiva: backup que não é testado regularmente não é backup

💡 Dica Prática

Crie um runbook de rollback para cada serviço: quem pode executar, qual o comando exato, quanto tempo leva, quais as dependências. E o mais importante: pratique o rollback regularmente em staging. No meio de um incidente às 3h da manhã, ninguém deveria estar lendo documentação pela primeira vez.

📊 Deploy Metrics (DORA)

“O que não se mede, não se melhora.” As métricas DORA (DevOps Research and Assessment) são o padrão da indústria para medir a performance de entrega de software. Validadas por 7 anos de pesquisa com milhares de organizações, correlacionam diretamente com sucesso organizacional.

💡 As 4 Métricas DORA

Dois pares de métricas que equilibram velocidade e estabilidade:

  • •Deployment Frequency — Com que frequência você faz deploy em produção (elite: múltiplas vezes/dia)
  • •Lead Time for Changes — Do commit ao deploy em produção (elite: < 1 hora)
  • •Change Failure Rate — Percentual de deploys que causam incidentes (elite: 0-15%)
  • •Mean Time to Recovery (MTTR) — Tempo para restaurar o serviço após falha (elite: < 1 hora)

🛠️ Como Implementar Métricas DORA

1

Instrumente o pipeline

Registre timestamps de cada deploy: quando o commit foi feito, quando passou pelo CI, quando chegou em produção.

2

Correlacione deploys com incidentes

Conecte seu sistema de alertas ao histórico de deploys. Qual deploy causou qual incidente? Quanto tempo levou para resolver?

3

Visualize e revise

Dashboard com as 4 métricas, atualizado automaticamente. Revise semanalmente com o time. Defina metas de melhoria incrementais.

📊 Dados: Impacto Organizacional

  • Times elite têm 3x menos burnout — Deploys frequentes e automatizados reduzem estresse
  • 2x mais chances de atingir metas comerciais com boas métricas DORA
  • 50% menos tempo gasto em retrabalho e correção de bugs em produção
  • Cultura de melhoria contínua — Métricas criam feedback loops que aceleram o aprendizado

💡 Dica Prática

Não tente melhorar as 4 métricas ao mesmo tempo. Comece com Deployment Frequency — ela força melhorias nas outras três. Para fazer deploy mais vezes, você precisa de testes melhores (menos falhas), pipeline mais rápido (menor lead time) e rollback confiável (menor MTTR).
