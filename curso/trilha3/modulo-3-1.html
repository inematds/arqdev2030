🧠 Arquitetura IA-First

🤖 O que é IA-First

IA-First não significa “usar IA em tudo”. Significa projetar o sistema em torno do modelo de IA como componente central, não como feature adicionado depois. A diferença é estrutural: a IA define a arquitetura, os fluxos de dados e as interfaces — tudo gira em torno dela.

💡 Conceito Principal

A diferença entre “sistema com IA” e “sistema IA-First”:

  • •Sistema com IA — CRUD tradicional que chama uma API de IA para enriquecer dados (ex: autocomplete)
  • •Sistema IA-First — O modelo é o core: sem ele, o produto não existe (ex: ChatGPT, Copilot, Midjourney)
  • •Implicação arquitetural — IA-First exige pensar em latência de inferência, custo por requisição, cache de respostas e fallbacks desde o design

📊 Caso Real: GitHub Copilot

O Copilot é um exemplo perfeito de arquitetura IA-First:

  • Core: Modelo Codex/GPT-4 — sem ele, o produto não existe
  • Latência: Precisa responder em <500ms para não atrapalhar o fluxo do dev
  • Custo: Cada sugestão custa tokens — GitHub perdeu US$20+/usuário/mês no início
  • Arquitetura: Streaming, cache agressivo, modelos menores para sugestões simples

✓ O que FAZER

  • ✓Definir SLAs específicos para inferência (latência, throughput)
  • ✓Projetar fallbacks para quando o modelo falhar ou demorar
  • ✓Calcular custo por requisição desde o dia 1

✗ O que NÃO fazer

  • ✗Tratar a API de IA como qualquer outro microserviço
  • ✗Ignorar que modelos são não-determinísticos
  • ✗Lançar sem rate limiting e controle de custos

💡 Dica Prática

Antes de começar qualquer projeto IA-First, faça a conta: “Se cada usuário fizer X requisições/dia, quanto vou gastar com tokens por mês?” Se o custo de inferência for maior que a receita por usuário, você tem um problema de modelo de negócio, não de arquitetura.

🔗 Padrões de Integração com LLMs

Existem múltiplas formas de integrar LLMs na arquitetura. Cada padrão tem trade-offs de custo, latência e controle. Escolher o padrão certo depende do caso de uso, do volume de requisições e do nível de personalização necessário.

💡 Padrões Arquiteturais para LLMs

  • •API Gateway + LLM — Chamada direta via API (OpenAI, Anthropic). Simples, mas sem controle sobre o modelo
  • •Proxy/Router de Modelos — Camada intermediária que roteia entre provedores (LiteLLM, OpenRouter)
  • •Self-hosted — Modelos open-source rodando na própria infra (Ollama, vLLM). Máximo controle, máximo custo operacional
  • •Hybrid — Modelos leves locais para tarefas simples, APIs externas para tarefas complexas

📊 Comparação de Abordagens

  • API externa: Latência ~500ms-2s, custo por token, zero ops, vendor lock-in
  • Self-hosted: Latência ~100-500ms, custo fixo de GPU, total controle, alta complexidade ops
  • Híbrido: Melhor custo-benefício para escala, complexidade média, flexibilidade máxima

💡 Dica Prática

Use um proxy de modelos (como LiteLLM) desde o início. Isso permite trocar de provedor sem mudar código, implementar fallback automático e monitorar custos centralizadamente. É a “camada anti-corrupção” do mundo de IA.

3

💰 Economia de Tokens

Em sistemas IA-First, tokens são a moeda principal. Cada prompt, cada resposta, cada retry consome tokens — e tokens custam dinheiro. Otimizar o uso de tokens é tão crítico quanto otimizar queries de banco de dados em sistemas tradicionais.

💡 Estratégias de Otimização

  • •Prompt caching — Reutilizar prefixos de prompt para reduzir tokens de input (Anthropic, OpenAI suportam nativamente)
  • •Semantic caching — Cachear respostas para perguntas semanticamente similares (não apenas idênticas)
  • •Model routing — Usar modelos baratos para tarefas simples, modelos caros só quando necessário
  • •Prompt compression — Reduzir o tamanho do contexto sem perder informação relevante

⚠️ Alerta: Custos Escalam Rápido

Uma startup com 10k usuários ativos fazendo 20 requisições/dia com GPT-4:

  • 200k requisições/dia × ~2k tokens/req = 400M tokens/dia
  • Custo estimado: US$4.000-12.000/dia dependendo do modelo
  • Sem otimização: US$120k-360k/mês só em inferência
  • Com cache + routing: Redução de 40-70% dos custos

💡 Dica Prática

Implemente budget alerts por usuário e por endpoint desde o MVP. Defina limites de tokens por requisição (max_tokens) e por usuário/dia. Um usuário abusivo pode consumir todo seu budget mensal em horas se não houver controle.

4

🛡️ Guardrails e Safety

LLMs são poderosos mas imprevisíveis. Sem guardrails, podem gerar conteúdo tóxico, vazar dados sensíveis do contexto, executar ações não autorizadas ou simplesmente alucinar. Guardrails são camadas de validação que protegem usuários e negócio.

💡 Camadas de Proteção

  • •Input validation — Filtrar prompts maliciosos (prompt injection, jailbreak) antes de enviar ao modelo
  • •Output validation — Verificar se a resposta está dentro do formato esperado e não contém dados sensíveis
  • •Scope limiting — Restringir as ações que o modelo pode executar (tools/functions disponíveis)
  • •Human-in-the-loop — Exigir aprovação humana para ações críticas (deletar dados, enviar emails, transações)

✓ O que FAZER

  • ✓Implementar detecção de prompt injection na entrada
  • ✓Usar structured outputs (JSON schema) para validar respostas
  • ✓Logar 100% das interações para auditoria

✗ O que NÃO fazer

  • ✗Confiar no system prompt como barreira de segurança
  • ✗Dar ao modelo acesso irrestrito a banco de dados ou APIs
  • ✗Exibir respostas do modelo sem sanitização no frontend

💡 Dica Prática

Use bibliotecas como Guardrails AI, NeMo Guardrails ou Rebuff para implementar validações de input/output. Não reinvente a roda — prompt injection é um campo de pesquisa ativo e evolui constantemente.

5

⚡ Latência e Streaming

LLMs são inerentemente lentos. Uma resposta de 500 tokens pode levar 3-10 segundos para gerar. Em UX, esperar 10 segundos por uma tela branca é inaceitável. Streaming é a solução: enviar tokens conforme são gerados, não esperar a resposta completa.

💡 Estratégias para Latência

  • •Server-Sent Events (SSE) — Streaming token a token via HTTP. Simples, amplamente suportado
  • •WebSockets — Conexão bidirecional para interações complexas (chat, voz)
  • •Speculative execution — Começar a gerar a resposta enquanto o usuário ainda digita
  • •Modelos menores como filtro — Usar modelo rápido para classificar antes de chamar o modelo pesado

📊 Métricas de Latência para IA

  • TTFB (Time to First Byte): Tempo até o primeiro token chegar — meta: <500ms
  • TPS (Tokens Per Second): Velocidade de geração — GPT-4: ~30-80 TPS, modelos locais: 10-50 TPS
  • E2E Latency: Tempo total da requisição — depende do tamanho da resposta
  • Percepção: Com streaming, usuário percebe latência de 300ms mesmo que a resposta leve 8s

💡 Dica Prática

Sempre use streaming por padrão. Não existe razão para esperar a resposta completa em interfaces de usuário. Para APIs máquina-a-máquina, avalie caso a caso — parsing de streaming JSON é mais complexo que esperar a resposta completa.

6

🔀 Fallback e Resiliência

APIs de IA falham. Modelos ficam lentos. Rate limits são atingidos. Sistemas IA-First precisam funcionar mesmo quando a IA não funciona. Isso exige uma estratégia de fallback bem definida: o que acontece quando o modelo principal não responde?

💡 Estratégias de Fallback

  • •Model fallback chain — GPT-4 → Claude → Llama local → resposta cached → mensagem genérica
  • •Circuit breaker — Parar de chamar um provedor que está falhando e redirecionar automaticamente
  • •Graceful degradation — Oferecer funcionalidade reduzida em vez de erro total
  • •Retry com backoff — Retentar com intervalos crescentes para erros transitórios (429, 503)

📊 Caso Real: OpenAI Outage

Em 2024, a OpenAI teve múltiplos outages que afetaram milhões de aplicações:

  • Apps sem fallback: Ficaram completamente fora do ar durante horas
  • Apps com fallback: Redirecionaram para Anthropic/modelos locais em segundos
  • Lição: Nunca dependa de um único provedor de IA em produção

💡 Dica Prática

Implemente um health check contínuo para cada provedor de IA. Mande um prompt de teste a cada 30 segundos e meça latência. Se a latência ultrapassar o threshold, ative o circuit breaker antes que os usuários percebam o problema.

🔎 Observabilidade de IA

Observabilidade em sistemas de IA é fundamentalmente diferente de sistemas tradicionais. Além de métricas de infra (CPU, memória, latência), você precisa monitorar qualidade das respostas, custos de tokens, drift de comportamento e feedback dos usuários.

💡 Pilares da Observabilidade de IA

  • •Traces de LLM — Rastrear cada requisição: prompt, resposta, tokens, latência, custo (Langfuse, Langsmith)
  • •Avaliação de qualidade — Métricas automáticas (relevância, groundedness) e feedback humano
  • •Dashboards de custo — Custo por usuário, por feature, por modelo, por dia
  • •Alertas de drift — Detectar quando a qualidade das respostas degrada (model drift, data drift)

🛠️ Stack de Observabilidade para IA

1

Logging estruturado

Logar prompt, resposta, modelo, tokens (input/output), latência e custo em formato JSON. Ferramenta: Langfuse, Helicone.

2

Métricas de negócio

Taxa de aceitação das sugestões, NPS de respostas, taxa de escalation para humano. Dashboard: Grafana, Datadog.

3

Avaliação contínua

Rodar evals automáticos periodicamente com dataset de referência. Comparar versões de prompts. Ferramenta: Promptfoo, Braintrust.

💡 Dica Prática

Implemente Langfuse ou Helicone no dia 1. São open-source e levam minutos para integrar. Sem observabilidade, você não sabe se seu sistema de IA está funcionando bem — sabe apenas que está funcionando.
