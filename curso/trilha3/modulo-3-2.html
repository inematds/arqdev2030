📚 RAG — Retrieval-Augmented Generation

RAG é o padrão mais importante para LLMs em produção. Em vez de treinar o modelo com seus dados, você busca informações relevantes em tempo real e injeta no prompt. Isso elimina alucinações, mantém dados atualizados e reduz custos de fine-tuning.

💡 Arquitetura RAG

  • •Indexação — Documentos são chunked, embedados e armazenados em vector database (Pinecone, Weaviate, pgvector)
  • •Retrieval — Query do usuário é embedada e busca os chunks mais similares (semantic search)
  • •Augmentation — Chunks relevantes são injetados no prompt como contexto
  • •Generation — LLM gera resposta baseada no contexto fornecido, não no conhecimento interno

📊 RAG em Números

  • 92% das empresas que usam LLMs em produção implementam alguma forma de RAG
  • Redução de 60-80% em alucinações comparado com LLM puro
  • Custo: 10-100x mais barato que fine-tuning para a maioria dos casos de uso
  • Atualização: Novos documentos disponíveis em minutos vs. semanas de retraining

✓ O que FAZER

  • ✓Investir tempo em chunking strategy — tamanho e overlap importam muito
  • ✓Implementar hybrid search (semântico + keyword) para melhor recall
  • ✓Adicionar reranking após retrieval para melhorar precisão

✗ O que NÃO fazer

  • ✗Chunks muito grandes (diluem relevância) ou muito pequenos (perdem contexto)
  • ✗Ignorar metadados na indexação (data, fonte, tipo)
  • ✗Assumir que top-k = resposta certa sem avaliar relevância

💡 Dica Prática

Comece com pgvector no seu PostgreSQL existente antes de adotar um vector database dedicado. Para a maioria dos casos (<1M documentos), pgvector é suficiente e elimina um componente de infra. Escale para Pinecone/Weaviate só quando necessário.

🔧 Function Calling / Tool Use

Function Calling permite que LLMs executem ações no mundo real — consultar APIs, buscar dados, criar registros, enviar emails. O modelo não executa diretamente: ele decide qual função chamar e com quais parâmetros, e seu código executa.

💡 Como Function Calling Funciona

  • •Definição — Você descreve funções disponíveis via JSON Schema (nome, descrição, parâmetros)
  • •Decisão — O modelo analisa a pergunta e decide se precisa chamar alguma função
  • •Execução — Seu código executa a função e retorna o resultado ao modelo
  • •Síntese — O modelo usa o resultado para compor a resposta final ao usuário

📊 Caso Real: Assistentes de Atendimento

Empresas como Klarna e Intercom usam function calling para:

  • Consultar pedidos: get_order_status(order_id) — modelo extrai o ID da conversa
  • Processar devoluções: create_return(order_id, reason) — com confirmação humana
  • Resultado Klarna: 2/3 do atendimento automatizado, resolução 25% mais rápida

💡 Dica Prática

Classifique suas tools em read-only (seguras) e write (perigosas). Tools read-only podem ser executadas automaticamente. Tools write devem ter confirmação ou aprovação. Nunca dê ao modelo uma tool “execute_sql” com acesso total ao banco.

🔄 Agent Loops

Agentes são LLMs que executam tarefas multi-step autonomamente. Em vez de uma única chamada prompt → resposta, o agente entra em um loop: pensa, age, observa, e repete até completar a tarefa. É o padrão mais poderoso — e o mais perigoso.

💡 Anatomia de um Agent Loop

  • •Observe — Receber o estado atual (mensagem do usuário, resultado da última ação)
  • •Think — Analisar o estado e decidir próximo passo (chain-of-thought)
  • •Act — Executar uma tool ou responder ao usuário
  • •Loop — Repetir até a tarefa estar completa ou atingir limite de iterações

⚠️ Riscos de Agentes Autônomos

  • Loop infinito: Agente fica preso tentando resolver algo que não consegue — sem limite de iterações, consome tokens indefinidamente
  • Ações destrutivas: Sem guardrails, agente pode deletar dados, enviar emails, fazer transações
  • Custo imprevisível: Cada iteração do loop consome tokens — tarefas complexas podem custar US$10+
  • Debugging difícil: Comportamento não-determinístico torna reprodução de bugs quase impossível

💡 Dica Prática

Sempre defina max_iterations, max_tokens_budget e timeout para qualquer agent loop. Um agente sem limites é um cartão de crédito sem limite nas mãos de um robô. Comece com limites conservadores e aumente conforme ganhar confiança.

📝 Prompt Engineering como Infra

Em sistemas de produção, prompts são infraestrutura crítica, não strings soltas no código. Precisam de versionamento, testes, rollback e monitoramento — exatamente como qualquer outro componente de infra.

💡 Prompt como Código

  • •Versionamento — Cada prompt é versionado (v1, v2, v3) com changelog
  • •Templates — Usar template engines (Jinja2, Handlebars) com variáveis
  • •Testes — Suite de evals automáticos rodando em CI/CD a cada mudança de prompt
  • •A/B testing — Comparar versões de prompt em produção com métricas reais

✓ O que FAZER

  • ✓Armazenar prompts em arquivos separados, não inline no código
  • ✓Criar dataset de avaliação com inputs e outputs esperados
  • ✓Implementar rollback instantâneo para versões anteriores

✗ O que NÃO fazer

  • ✗Hardcodar prompts como strings no código
  • ✗Mudar prompts em produção sem testes
  • ✗Ignorar que mudança de modelo quebra prompts existentes

💡 Dica Prática

Use Promptfoo para criar suites de teste para seus prompts. Defina critérios de aceitação (contém X, não contém Y, formato JSON válido) e rode no CI a cada PR que altera prompts. Prompt regression é tão real quanto code regression.

⚖️ Fine-tuning vs Prompting vs RAG

A pergunta mais comum: “Devo fazer fine-tuning, RAG, ou só melhorar o prompt?” A resposta depende do tipo de conhecimento, do volume de dados e do budget. Na maioria dos casos, a ordem correta é: prompt engineering → RAG → fine-tuning.

💡 Quando Usar Cada Abordagem

  • •Prompt Engineering — Conhecimento genérico, formato específico, baixo volume. Custo: zero. Tempo: horas
  • •RAG — Dados proprietários, atualizados frequentemente, base de conhecimento grande. Custo: médio. Tempo: dias
  • •Fine-tuning — Estilo/tom específico, tarefas muito especializadas, performance crítica. Custo: alto. Tempo: semanas
  • •Combinação — Fine-tuned model + RAG é o estado da arte, mas só justifica em escala

📊 Comparação Direta

  • Prompt: 0 dados necessários, implementação em minutos, sem custo adicional
  • RAG: Documentos existentes, implementação em dias, custo de vector DB + embeddings
  • Fine-tuning: 1k-100k exemplos anotados, implementação em semanas, US$100-10k+ por training run

💡 Dica Prática

Regra de ouro: “Se prompt engineering resolve 80% do problema, não faça RAG. Se RAG resolve 80%, não faça fine-tuning.” Cada salto de complexidade multiplica custo e tempo de manutenção. Só avançe quando o nível anterior comprovadamente não atende.

🔒 Segurança de IA

Segurança em sistemas de IA vai além de guardrails. Envolve proteção de dados de treinamento, prevenção de data leakage, compliance regulatório e defesa contra ataques específicos de IA como prompt injection, data poisoning e model extraction.

💡 Superfície de Ataque de IA

  • •Prompt Injection — Usuário manipula o prompt para extrair dados ou mudar comportamento
  • •Data Leakage — Modelo revela dados sensíveis do contexto (PII, segredos)
  • •Indirect Injection — Dados maliciosos injetados via RAG (documentos envenenados)
  • •Model Extraction — Ataques para replicar ou roubar o comportamento do modelo

✓ Defesas Essenciais

  • ✓Sanitizar inputs — Remover/escapar instruções maliciosas antes do prompt
  • ✓PII detection — Filtrar dados pessoais do output antes de exibir
  • ✓Rate limiting por usuário — Limitar requisições para prevenir abuso

✗ Erros Comuns

  • ✗Colocar secrets no system prompt (modelo pode revelar)
  • ✗Confiar que o modelo “nunca vai” fazer algo proibido
  • ✗Indexar documentos sensíveis no RAG sem controle de acesso

💡 Dica Prática

Implemente OWASP Top 10 for LLM Applications como checklist de segurança. Trate o modelo como um “usuário não confiável” — valide tudo que entra e tudo que sai, assim como faria com input de um formulário web.

🚀 Escalando Sistemas com IA

Escalar sistemas de IA é fundamentalmente diferente de escalar CRUD tradicionais. Você não pode simplesmente adicionar instâncias — cada requisição de IA é computacionalmente cara, e o gargalo geralmente é o provedor, não sua infra.

💡 Estratégias de Escala para IA

  • •Queue-based processing — Desacoplar requisições do processamento. Usuário envia, recebe quando pronto
  • •Multi-provider load balancing — Distribuir entre OpenAI, Anthropic, modelos locais
  • •Batch processing — Agrupar requisições similares para processar em lote (mais barato)
  • •Edge inference — Modelos pequenos rodando no edge para requisições simples

🛠️ Escala Progressiva

1

MVP (0-1k usuários)

API direta para um provedor. Sem cache. Sem filas. Foco em validar o produto.

2

Growth (1k-50k usuários)

Semantic cache, proxy de modelos, rate limiting, multi-provider fallback. Budget alerts.

3

Scale (50k+ usuários)

Modelos self-hosted para workloads previsíveis. Batch processing. Edge inference. Fine-tuning para reduzir tokens.

💡 Dica Prática

O maior erro de escala em IA é tratar todas as requisições como iguais. Classifique por prioridade e complexidade: requisições simples vão para modelos rápidos/baratos, requisições complexas para modelos poderosos. Isso pode reduzir custos em 50-70%.
