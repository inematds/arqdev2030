📈 Vertical vs Horizontal Scaling

A primeira grande decisão de escala é escolher entre scale up (máquina mais potente) e scale out (mais máquinas). Vertical scaling tem um teto físico — não existe CPU infinita. Horizontal scaling escala teoricamente sem limite, mas exige que a aplicação seja stateless e distribuída.

💡 Conceito Principal

Entenda as duas abordagens e quando usar cada uma:

  • •Vertical (Scale Up) — Mais CPU, RAM, disco na mesma máquina. Simples, sem mudança de código. Limite: hardware disponível
  • •Horizontal (Scale Out) — Mais instâncias atrás de um load balancer. Escala infinita, mas exige stateless design
  • •Híbrido — Vertical para banco de dados (mais fácil), horizontal para aplicação (mais flexível)

📊 Caso Real: Stack Overflow vs Twitter

Duas abordagens opostas, ambas servindo milhões de usuários:

  • Stack Overflow — 9 servidores web atendem 1.3 bilhão de pageviews/mês. Vertical scaling + otimização extrema de código (.NET + SQL Server altamente tunado)
  • Twitter — Milhares de servidores processando 500 milhões de tweets/dia. Horizontal scaling com microsserviços em JVM
  • Lição — Não existe resposta universal. O contexto (tipo de carga, padrão de acesso, budget) define a melhor abordagem

✓ O que FAZER

  • ✓Começar com vertical scaling — é mais simples e barato inicialmente
  • ✓Projetar a aplicação como stateless desde o dia 1 para facilitar horizontal futuro
  • ✓Medir antes de escalar — talvez o gargalo seja código, não infra

✗ O que NÃO fazer

  • ✗Guardar sessões em memória local — impede horizontal scaling
  • ✗Escalar horizontalmente sem observabilidade — mais instâncias = mais complexidade
  • ✗Assumir que “mais servidores = mais rápido” — Amdahl’s Law limita o ganho

💡 Dica Prática

Regra de ouro: escale verticalmente até que fique caro demais, depois escale horizontalmente. Um servidor com 64 cores e 256GB de RAM custa muito menos para operar do que 10 servidores pequenos com load balancer, service discovery e todo o overhead de sistemas distribuídos.

⚖️ Load Balancing

Load balancing é a distribuição inteligente de tráfego entre múltiplas instâncias de uma aplicação. É o componente que viabiliza horizontal scaling, garantindo que nenhum servidor fique sobrecarregado enquanto outros estão ociosos.

💡 Algoritmos de Balanceamento

Os principais algoritmos e quando usar cada um:

  • •Round Robin — Distribui sequencialmente. Simples, funciona quando todos os servidores são iguais
  • •Least Connections — Envia para quem tem menos conexões ativas. Ideal para requests de duração variável
  • •IP Hash — Mesmo cliente vai sempre para o mesmo servidor. Útil para sticky sessions
  • •Weighted — Servidores mais potentes recebem mais tráfego. Ótimo para infra heterogênea

📊 Caso Real: Netflix e Zuul

A Netflix desenvolveu o Zuul, um gateway/load balancer que processa bilhões de requests por dia:

  • Layer 7 load balancing — Roteia com base no conteúdo da requisição (URL, headers, parâmetros)
  • Canary deployments — Envia 1% do tráfego para versão nova, valida, e gradualmente aumenta
  • Resiliência — Circuit breakers integrados removem automáticamente servidores com falha

✓ O que FAZER

  • ✓Implementar health checks para remover instâncias com falha automaticamente
  • ✓Usar TLS termination no load balancer para reduzir carga nos backends
  • ✓Configurar connection draining para deploys sem downtime

✗ O que NÃO fazer

  • ✗Depender de sticky sessions quando a aplicação deveria ser stateless
  • ✗Ter um único load balancer sem redundancy — ponto único de falha
  • ✗Ignorar métricas de latência do próprio load balancer

💡 Dica Prática

Para a maioria das aplicações, um ALB (Application Load Balancer) managed na cloud é suficiente. Só considere soluções customizadas (Envoy, HAProxy, Nginx) quando precisar de lógica de roteamento que o ALB não suporta.

🗃️ Database Scaling

O banco de dados é quase sempre o primeiro gargalo de sistemas em crescimento. Diferente da aplicação, bancos de dados têm estado, o que torna horizontal scaling significativamente mais complexo. As estratégias envolvem read replicas, sharding, particionamento e CQRS.

💡 Estratégias de Database Scaling

  • •Read Replicas — Cópias do banco para leitura. Write no primário, read nos secundários. 80% dos workloads são leitura
  • •Sharding — Dividir dados entre múltiplos bancos por chave (user_id, região). Escala writes, mas complica queries cross-shard
  • •Particionamento — Dividir tabelas grandes por range (data) ou hash. Melhora performance sem mudar a aplicação
  • •CQRS — Bancos separados para leitura e escrita. Otimiza cada lado independentemente

📊 Caso Real: Instagram

O Instagram escalou o PostgreSQL para bilhões de fotos:

  • Fase 1: Vertical scaling do PostgreSQL até chegar ao limite do hardware
  • Fase 2: Read replicas para separar tráfego de leitura (feeds, buscas)
  • Fase 3: Sharding por user_id — cada shard com um subconjunto de usuários
  • Resultado: 2 bilhões de usuários ativos com PostgreSQL — não precisaram de NoSQL

✓ O que FAZER

  • ✓Começar com read replicas — resolve 80% dos problemas de escala de DB
  • ✓Escolher a shard key com cuidado — redistribuir depois é extremamente caro
  • ✓Implementar connection pooling (PgBouncer, ProxySQL) antes de escalar

✗ O que NÃO fazer

  • ✗Shardar prematuramente — a complexidade operacional é enorme
  • ✗Ignorar replication lag em read replicas — pode causar dados inconsistentes
  • ✗Trocar de banco “porque NoSQL escala melhor” sem entender o trade-off

💡 Dica Prática

A ordem de escala de banco mais pragmática: 1) Otimizar queries e índices → 2) Connection pooling → 3) Read replicas → 4) Caching → 5) Particionamento → 6) Sharding. A maioria dos sistemas nunca precisa passar do passo 4.

⚡ Caching em Escala

Caching é a arma mais poderosa para performance e escalabilidade. Um cache hit evita uma query no banco, uma chamada de API, ou um cálculo caro. Mas caching mal implementado causa bugs sutis — dados desatualizados, cache stampede e inconsistências.

💡 Camadas de Cache

  • •Browser Cache — Headers HTTP (Cache-Control, ETag). Mais rápido possível — zero latência de rede
  • •CDN Cache — Conteúdo estático e dinâmico na edge. Cloudflare, CloudFront, Fastly
  • •Application Cache — Redis/Memcached para dados frequentemente acessados. Session, cart, user profile
  • •Database Cache — Query cache, buffer pool. Gerenciado pelo próprio banco

⚠️ Problemas Clássicos de Cache

  • Cache Stampede — Quando o cache expira e milhares de requests vão simultâneamente ao banco. Solução: locking ou probabilistic early expiration
  • Cache Invalidation — “As duas coisas mais difíceis em CS: cache invalidation e nomear coisas”. Use TTL + event-driven invalidation
  • Cold Start — Cache vazio após deploy ou crash. Implemente cache warming para dados críticos
  • Stale Data — Dados desatualizados servidos pelo cache. Defina TTLs adequados ao domínio

💡 Dica Prática

Use a estratégia Cache-Aside (Lazy Loading) como padrão: leia do cache; se miss, leia do banco e popule o cache. Para dados que mudam frequentemente, combine com Write-Through (escreve no cache e no banco ao mesmo tempo) para manter consistência.

📦 Queue-Based Load Leveling

Filas de mensagens absorvem picos de tráfego transformando processamento síncrono em assíncrono. Em vez de o usuário esperar o processamento completo, a requisição é enfileirada e processada no ritmo que o sistema suporta.

💡 Padrões com Filas

  • •Work Queue — Tarefas distribuídas entre workers. Cada mensagem processada por um único consumer
  • •Pub/Sub — Mensagem entregue a todos os subscribers. Ideal para eventos e notificações
  • •Dead Letter Queue — Mensagens que falharam vão para fila separada para análise e retry
  • •Priority Queue — Mensagens com prioridade são processadas primeiro (ex: pagamentos antes de e-mails)

📊 Caso Real: Uber

A Uber processa milhões de eventos por segundo usando Apache Kafka:

  • Localização GPS — Motoristas enviam posição a cada segundo. Processamento síncrono seria impossível
  • Matching — Eventos de solicitação de corrida são enfileirados e processados por algoritmo de matching
  • Surge pricing — Eventos de demanda alimentam cálculo de preço dinâmico via streams

💡 Dica Prática

Comece com SQS ou RabbitMQ para work queues simples. Migre para Kafka quando precisar de event streaming, replay de eventos ou múltiplos consumers lendo a mesma mensagem. Kafka é poderoso mas tem custo operacional alto — não use para fila de e-mail.

🌐 CDN e Edge Computing

CDNs e edge computing levam a computação para perto do usuário. Em vez de todas as requisições viajarem até um datacenter central, conteúdo e lógica são distribuídos em centenas de PoPs (Points of Presence) ao redor do mundo.

💡 CDN vs Edge Computing

  • •CDN Tradicional — Cache de conteúdo estático (imagens, CSS, JS, vídeos). Push ou pull. Cloudflare, CloudFront, Akamai
  • •Edge Computing — Execução de código na borda. Lógica de negócio rodando em 300+ PoPs. Cloudflare Workers, Vercel Edge, Deno Deploy
  • •Edge + DB — Bancos distribuídos na edge (Turso, Neon, PlanetScale) completam o ecossistema

📊 Impacto de CDN em Números

  • 50-70% de redução na latência para usuários distantes do servidor origin
  • 60-80% de redução no tráfego para o servidor origin com cache adequado
  • Amazon — Cada 100ms de latência adicional custa 1% nas vendas
  • Google — Páginas que demoram >3s perdem 53% dos visitantes mobile

💡 Dica Prática

Coloque tudo que puder atrás de CDN: assets estáticos, páginas pré-renderizadas, respostas de API com cache headers. Para aplicações globais, considere Cloudflare Workers como primeiro ponto de contato — roda JavaScript na edge com cold start de ~0ms.

📋 Scaling Checklist

Antes de escalar, organize e priorize. Este checklist consolida as estratégias deste módulo em um guia prático para avaliar a prontidão do sistema para crescimento.

🛠️ Checklist de Escala em 6 Fases

1

Medir e Diagnosticar

Instrumente o sistema com métricas de latência, throughput e erro. Identifique o gargalo real antes de otimizar qualquer coisa.

2

Otimizar Código e Queries

N-plus-1 queries, índices faltando, serialização ineficiente. Frequentemente, otimizar código elimina a necessidade de mais infra.

3

Adicionar Cache

Redis/Memcached para dados quentes, CDN para assets, browser cache para conteúdo estático. Maior impacto com menor esforço.

4

Escalar Verticalmente

Mais CPU, RAM, SSD. Simples, sem mudança de código. Funciona até o limite do hardware e do orçamento.

5

Escalar Horizontalmente

Load balancer + múltiplas instâncias. Requer stateless design. Auto-scaling para lidar com picos.

6

Distribuir e Especializar

Sharding, CQRS, microsserviços, edge computing. Máxima escala, máxima complexidade. Só quando necessário.

💡 Dica Prática

Imprima este checklist e siga a ordem. A maioria dos times pula direto para a fase 5 (horizontal scaling) quando o problema está na fase 2 (uma query sem índice fazendo full table scan). Disciplina de diagnóstico vale mais que infraestrutura cara.
