🔍 Profiling

“Premature optimization is the root of all evil.” — Donald Knuth. Antes de otimizar qualquer coisa, você precisa medir. Profiling é o processo de identificar onde o sistema realmente gasta tempo e recursos, substituindo intuição por dados.

💡 Tipos de Profiling

  • •CPU Profiling — Identifica funções que consomem mais CPU. Ferramentas: pprof (Go), py-spy (Python), async-profiler (Java)
  • •Memory Profiling — Detecta memory leaks e alocações excessivas. Heap snapshots, GC analysis
  • •I/O Profiling — Mede tempo gasto em disco, rede e banco de dados. Geralmente o maior gargalo
  • •Distributed Tracing — Rastreia requisições através de múltiplos serviços. Jaeger, Zipkin, OpenTelemetry

📊 Caso Real: LinkedIn

O LinkedIn usou profiling contínuo para reduzir custos de infraestrutura:

  • Problema: Custos de cloud crescendo mais rápido que a receita
  • Descoberta via profiling: 30% do CPU era gasto em serialização JSON desnecessária
  • Solução: Migração para Protocol Buffers em hot paths
  • Resultado: 40% de redução no uso de CPU, economia de milhões de dólares/ano

✓ O que FAZER

  • ✓Implementar continuous profiling em produção (low overhead)
  • ✓Usar flame graphs para visualizar onde o tempo é gasto
  • ✓Medir p99 e p95, não apenas média — a média esconde outliers

✗ O que NÃO fazer

  • ✗Otimizar com base em “acho que esse trecho é lento”
  • ✗Profilear apenas em ambiente de dev — produção tem padrões diferentes
  • ✗Ignorar GC pauses que podem causar spikes de latência

💡 Dica Prática

Adote OpenTelemetry como padrão de instrumentação. É vendor-neutral, suporta traces, metrics e logs, e tem SDKs para todas as linguagens. Conecte ao Grafana + Tempo para visualização. Custo zero de vendor lock-in.

🗃️ Otimização de Queries

Queries mal escritas são a causa número 1 de problemas de performance. Uma única query sem índice pode derrubar um sistema inteiro. Dominar EXPLAIN, entender índices e eliminar N+1 é obrigatório para qualquer desenvolvedor.

💡 Técnicas de Otimização

  • •EXPLAIN ANALYZE — Mostra o plano de execução real da query. Seq Scan em tabela grande = problema
  • •Índices compostos — Ordem das colunas importa. (status, created_at) ≠ (created_at, status)
  • •N+1 Queries — 1 query para listar + N queries para detalhes. Use JOINs ou eager loading
  • •Paginação eficiente — Cursor-based pagination (WHERE id > last_id) em vez de OFFSET/LIMIT

⚠️ O Problema do N+1

O N+1 é tão comum e destrutivo que merece destaque especial:

  • Exemplo: Listar 100 pedidos com dados do cliente = 1 query (pedidos) + 100 queries (cliente de cada pedido)
  • Impacto: 101 queries quando 1 JOIN resolveria. Em produção com 10k pedidos = 10.001 queries
  • Detecção: Query logging, APM tools, ou gems/packages como bullet (Rails), nplusone (Django)
  • Solução: Eager loading, JOINs, ou DataLoader pattern (GraphQL)

💡 Dica Prática

Ative slow query log com threshold de 100ms. Revise semanalmente as top 10 queries mais lentas. Frequentemente, adicionar um único índice transforma uma query de 5 segundos em 5 milissegundos — 1000x de melhoria sem mudar código.

🎨 Otimização de Assets

Assets (imagens, CSS, JS, fontes) representam 60-80% do peso total de uma página web. Otimizar assets é frequentemente a forma mais rápida de melhorar performance percebida pelo usuário, com impacto direto em conversão e SEO.

💡 Estratégias de Otimização

  • •Imagens — WebP/AVIF (30-50% menor que JPEG), lazy loading, responsive images (srcset), image CDN (Cloudinary, imgix)
  • •JavaScript — Tree shaking, code splitting, lazy imports. Bundle analyzer para detectar dependências pesadas
  • •CSS — PurgeCSS remove CSS não utilizado. Critical CSS inline no <head>. Resto carregado async
  • •Fontes — font-display: swap, subset de caracteres, self-hosting em vez de Google Fonts para TTFB

📊 Impacto em Números

  • Pinterest — Reduziu peso da página em 40%, sign-ups aumentaram 15%
  • Walmart — Cada 1 segundo de melhoria no load time = 2% mais conversão
  • BBC — Cada segundo adicional de carregamento perde 10% dos usuários
  • Google — Core Web Vitals (LCP, FID, CLS) são fator de ranking desde 2021

💡 Dica Prática

Execute Lighthouse no Chrome DevTools e foque primeiro nos itens de maior impacto. Normalmente, converter imagens para WebP e adicionar lazy loading resolve 50% dos problemas com 5 minutos de trabalho.

🖥️ Server-Side Performance

Performance no servidor envolve como o backend processa requisições: concorrência, gerenciamento de memória, I/O, serialização e computação. Escolhas de linguagem, runtime e patterns arquiteturais impactam diretamente o throughput.

💡 Áreas de Otimização Server-Side

  • •Concorrência — Async I/O (Node.js, Go goroutines), thread pools, reactive streams. Evite blocking operations
  • •Connection Pooling — Reutilize conexões com banco, HTTP clients, Redis. Criar conexão é caro (TCP handshake + TLS)
  • •Serialização — JSON é legível mas lento. Protocol Buffers, MessagePack para hot paths internos
  • •Compressão — Gzip/Brotli para respostas HTTP. Brotli é 15-25% melhor que Gzip em tamanho

📊 Caso Real: Discord

O Discord migrou de Go para Rust em serviços críticos por performance:

  • Problema: Go’s garbage collector causava spikes de latência a cada 2 minutos
  • Solução: Reescrita do Read States service em Rust (sem GC)
  • Resultado: Latência p99 caiu de 130ms para 10ms. Sem spikes de GC
  • Lição: A escolha de linguagem importa quando performance é requisito crítico

💡 Dica Prática

Antes de trocar de linguagem, otimize o que já tem. A maioria dos problemas de performance é algorítmica (O(n²) que deveria ser O(n)) ou de I/O (queries síncronas que poderiam ser paralelas), não da linguagem em si.

5

🎨 Frontend Performance

O usuário não se importa se o servidor respondeu em 10ms se a página demora 5 segundos para ficar interativa. Performance percebida é tão importante quanto performance real. Core Web Vitals do Google medem exatamente isso.

💡 Core Web Vitals

  • •LCP (Largest Contentful Paint) — Tempo até o maior elemento visível carregar. Meta: < 2.5s
  • •INP (Interaction to Next Paint) — Responsividade a interações do usuário. Meta: < 200ms
  • •CLS (Cumulative Layout Shift) — Estabilidade visual — nada deve “pular” na tela. Meta: < 0.1

✓ Quick Wins

  • ✓Preload de recursos críticos (fontes, hero image, CSS)
  • ✓Skeleton screens em vez de spinners — performance percebida
  • ✓Code splitting por rota — carregue só o JS necessário para a página atual

✗ Performance Killers

  • ✗Bundle único de 2MB+ de JavaScript carregado na home
  • ✗Imagens sem dimensões definidas causando layout shift
  • ✗Third-party scripts bloqueando render (analytics, chat widgets)

💡 Dica Prática

Configure Web Vitals monitoring em produção com dados reais de usuários (RUM — Real User Monitoring). Lighthouse mede em ambiente controlado; RUM mostra a realidade em dispositivos lentos e conexões 3G.

6

📦 Redução de Payload

Cada byte transferido custa latência, banda e bateria (em mobile). Reduzir payload de APIs, respostas HTTP e transfers de dados é uma das otimizações com melhor custo-benefício, especialmente para usuários em redes lentas.

💡 Técnicas de Redução

  • •Sparse Fieldsets — Retorne apenas os campos solicitados (GraphQL faz isso nativamente; REST com ?fields=id,name)
  • •Compressão — Brotli para respostas HTTP (suportado por todos os browsers modernos). 15-25% menor que Gzip
  • •Paginação — Nunca retorne listas completas. Cursor-based para grandes datasets
  • •Delta Updates — Envie apenas o que mudou (ETags, If-Modified-Since, WebSockets para real-time)

💡 Dica Prática

Analise suas APIs com Chrome DevTools > Network tab. Ordene por tamanho. Frequentemente, APIs retornam 10x mais dados do que o frontend usa. Um endpoint que retorna o objeto User com 50 campos quando o frontend usa 5 é desperdício puro.

🔧 Performance Testing

“Hope is not a strategy.” Testar performance sistematicamente é a única forma de garantir que o sistema vai aguentar a carga de produção. Load tests, stress tests e chaos engineering revelam problemas que code review nunca encontraria.

💡 Tipos de Teste de Performance

  • •Load Test — Carga esperada por período sustentado. Valida se o sistema atende SLAs sob uso normal
  • •Stress Test — Carga acima do esperado. Encontra o ponto de ruptura e verifica degradação graciosa
  • •Spike Test — Picos súbitos de tráfego. Valida auto-scaling e comportamento sob carga instantânea
  • •Soak Test — Carga moderada por tempo prolongado. Detecta memory leaks e degradação gradual

🛠️ Ferramentas Recomendadas

1

k6 (Grafana)

Testes escritos em JavaScript, fácil de versionar. Suporte a protocolos HTTP, WebSocket, gRPC. Integração nativa com Grafana Cloud.

2

Locust

Testes em Python, distribuído, UI web embutida. Excelente para cenários complexos com lógica condicional. Open source.

3

Artillery

YAML-based, simples para começar. Suporta HTTP, WebSocket e Socket.io. Bom para CI/CD pipelines com testes rápidos.

💡 Dica Prática

Integre testes de performance no CI/CD. Um load test de 2 minutos em cada PR que toca endpoints críticos previne regressões. Use k6 com thresholds: se p95 > 200ms, o pipeline falha. Prevenção é 100x mais barata que correção em produção.
