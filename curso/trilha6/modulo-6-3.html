🏢 Modelos de Multi-tenancy

Multi-tenancy é a capacidade de um único sistema servir múltiplos clientes (tenants) de forma isolada. A escolha do modelo de tenancy impacta diretamente custo, segurança, performance e complexidade operacional.

💡 Os 3 Modelos Fundamentais

  • •Banco Compartilhado (Shared DB) — Todos os tenants na mesma tabela, diferenciados por tenant_id. Mais barato, menor isolamento
  • •Schema por Tenant — Mesmo banco, schemas separados. Bom equilíbrio entre custo e isolamento
  • •Banco por Tenant — Cada cliente tem seu próprio banco. Máximo isolamento, custo mais alto, mais complexo para gerenciar

📊 Caso Real: Salesforce vs Slack

Dois gigantes SaaS com abordagens opostas:

  • Salesforce — Shared DB com metadata-driven architecture. Um único banco Oracle serve 150k+ empresas. Custo por tenant é mínimo
  • Slack — Shard por workspace. Cada workspace tem seus dados em shards dedicados. Facilita compliance e data residency
  • Lição — Não existe modelo universalmente melhor. Requisitos de compliance, custo e escala definem a escolha

✓ O que FAZER

  • ✓Começar com shared DB + tenant_id — mais simples e barato para começar
  • ✓Usar Row-Level Security (RLS) do PostgreSQL para garantir isolamento a nível de banco
  • ✓Projetar para migrar entre modelos no futuro sem reescrever a aplicação

✗ O que NÃO fazer

  • ✗Confiar apenas na aplicação para isolamento — um bug expõe dados de outros tenants
  • ✗Escolher banco por tenant com 10 clientes — overhead operacional injustificável
  • ✗Ignorar noisy neighbor — um tenant pesado afetando performance de todos

💡 Dica Prática

Use PostgreSQL RLS (Row-Level Security) como rede de segurança. Mesmo que a aplicação filtre por tenant_id, o RLS garante que uma query sem filtro nunca retorne dados de outro tenant. Segurança em profundidade é obrigatória em multi-tenancy.

🔒 Isolamento de Dados

Isolamento de dados é o aspecto mais crítico de multi-tenancy. Um vazamento de dados entre tenants pode destruir a confiança no produto e resultar em multas milionárias (LGPD, GDPR). Isolamento precisa ser implementado em múltiplas camadas.

💡 Camadas de Isolamento

  • •Banco de Dados — RLS, schemas separados, ou bancos dedicados. Primeira e mais importante barreira
  • •Aplicação — Middleware que injeta tenant_id em toda query. Contexto de tenant em cada request
  • •Storage — Buckets separados ou prefixos por tenant no S3/GCS. Prevenção de acesso cruzado
  • •Cache — Namespace por tenant no Redis. Chave: tenant:{id}:user:{user_id} para evitar colisões
  • •Logs e Métricas — Tenant ID como label obrigatório em todos os logs e traces

⚠️ Riscos de Vazamento Mais Comuns

  • Query sem WHERE tenant_id — Retorna dados de todos os tenants. Mais comum do que parece, especialmente em queries manuais
  • Cache poisoning — Dados de um tenant cacheados e servidos para outro por chave de cache genérica
  • Background jobs — Jobs que processam dados sem contexto de tenant, misturando resultados
  • API endpoints sem validação — /api/orders/123 acessado por tenant que não é dono do pedido 123

💡 Dica Prática

Crie testes automatizados específicos para isolamento: crie 2 tenants, insira dados em ambos, e valide que cada um só vê seus próprios dados em TODA operação (list, get, search, export). Rode esses testes no CI/CD em cada PR.

💰 Billing e Subscription

O modelo de billing define como o produto gera receita. Em SaaS, isso geralmente envolve planos de assinatura, cobrança por uso, ou modelos híbridos. A arquitetura de billing precisa ser robusta, auditável e flexível para mudanças de preço.

💡 Modelos de Monetização SaaS

  • •Flat Rate — Preço fixo mensal por plano (Starter, Pro, Enterprise). Simples de entender e implementar
  • •Per Seat — Cobrança por usuário ativo. Slack, GitHub, Notion. Escala com adoção dentro da empresa
  • •Usage-Based — Pague pelo que usar. AWS, Twilio, Stripe. Alinha custo com valor recebido
  • •Híbrido — Base fixa + variável por uso. Tendência crescente. Ex: plano base + cobrança por API calls excedentes

📊 Caso Real: Stripe como Infraestrutura de Billing

A maioria dos SaaS modernos usa Stripe Billing como plataforma:

  • Stripe Billing — Gerencia subscriptions, invoices, proration, trials e downgrades
  • Webhooks — Events como invoice.paid, subscription.canceled disparam ações no backend
  • Metered Billing — Reporte uso via API, Stripe calcula e cobra no fim do período
  • Lição — Não reinvente billing. Use Stripe, Paddle ou similar. Billing customizado é uma armadilha

💡 Dica Prática

Implemente billing como um serviço separado que se comunica por eventos. Quando o billing diz “plano mudou para Pro”, o sistema de features atualiza as permissões. Essa separação permite mudar de Stripe para outro provedor sem tocar na lógica de negócio.

🏷️ Feature Flags por Tenant

Feature flags permitem ativar funcionalidades seletivamente por tenant, plano, ou segmento. São essenciais para diferenciar planos (Free vs Pro), beta testing com clientes específicos e rollout gradual de features novas.

💡 Tipos de Feature Flags

  • •Plan-based — Features atreladas ao plano. Free tem X, Pro tem Y, Enterprise tem Z
  • •Tenant-specific — Feature habilitada para tenants específicos (beta testers, parceiros estratégicos)
  • •Percentage rollout — Feature ativa para X% dos tenants. Aumenta gradualmente monitorando métricas
  • •Kill switch — Desabilitar feature instantâneamente em produção sem deploy

✓ O que FAZER

  • ✓Usar ferramentas como LaunchDarkly, Unleash ou Flagsmith em vez de if/else hardcoded
  • ✓Verificar flags tanto no frontend quanto no backend — esconder UI não basta
  • ✓Definir ciclo de vida para cada flag — flags permanentes e temporárias

✗ O que NÃO fazer

  • ✗Acumular centenas de flags sem limpeza — vira dívida técnica
  • ✗Validar acesso a features só no frontend — API direta burla qualquer validação UI
  • ✗Usar feature flags como sistema de permissões — são coisas diferentes

💡 Dica Prática

Crie uma matriz de features por plano como fonte de verdade. Mantenha essa matriz em configuração (não no código) para que o time de produto possa ajustar sem deploy. Exemplo: features.yaml com plan_id → lista de features habilitadas.

🚀 Onboarding Automatizado

Em SaaS, o onboarding precisa ser self-service e instantâneo. O cliente cria conta, escolhe plano, configura o workspace e começa a usar — tudo sem intervenção humana. Quanto mais atrito no onboarding, maior o churn antes da primeira experiência de valor.

💡 Pipeline de Onboarding

  • •Tenant Provisioning — Criar registros no banco, schema/namespace, configurações padrão, usuário admin
  • •DNS e Subdomínio — tenant.app.com ou domínio customizado. Wildcard DNS + TLS automático (Let’s Encrypt)
  • •Seed Data — Dados iniciais para que o tenant veja valor imediatamente (templates, exemplos, wizards)
  • •Integrações — OAuth setup, import de dados, conexão com ferramentas existentes

📊 Caso Real: Notion

O Notion é referência em onboarding self-service:

  • Tempo de valor: Workspace funcional em <30 segundos após sign-up
  • Templates: Conteúdo pré-populado que mostra o potencial da ferramenta imediatamente
  • Progressive disclosure: Features avançadas aparecem conforme o uso, sem sobrecarregar no início
  • Resultado: Product-led growth com milhões de usuários sem time de vendas tradicional

💡 Dica Prática

Meça o Time to First Value (TTFV) — quanto tempo leva do sign-up até o usuário realizar a primeira ação de valor. Se o TTFV é > 5 minutos, você está perdendo clientes. Reduza atrito: login social, wizard de setup, templates pré-configurados.

📈 Scaling Multi-tenant

Escalar um sistema multi-tenant traz desafios únicos: noisy neighbors (um tenant consumindo recursos excessivos), distribuição desigual de carga entre tenants, e a necessidade de escalar sem afetar a experiência de outros clientes.

💡 Estratégias de Scaling Multi-tenant

  • •Rate Limiting por Tenant — Limites de API calls por plano. Previne abuso e protege outros tenants
  • •Tenant Tiering — Tenants grandes em infra dedicada, pequenos compartilhados. Custo proporcional ao tamanho
  • •Queue Isolation — Filas separadas por prioridade de tenant. Enterprise não espera na fila do Free
  • •Shard Migration — Mover tenants entre shards conforme crescem, sem downtime

🛠️ Combatendo o Noisy Neighbor

1

Monitorar uso por tenant

Métricas de CPU, memória, I/O e API calls segmentadas por tenant_id. Dashboards que mostram os top 10 consumers.

2

Aplicar rate limiting e throttling

Limites por plano: Free = 100 req/min, Pro = 1000 req/min, Enterprise = custom. Token bucket algorithm.

3

Isolar ou migrar tenants problemáticos

Quando um tenant cresce demais para o pool compartilhado, migre para infra dedicada. Automatize essa migração.

💡 Dica Prática

Implemente fair-share scheduling: mesmo que um tenant tenha rate limit alto, ele não pode monopolizar 100% dos recursos compartilhados. Reserve capacidade mínima para cada tenant ativo, garantindo SLA básico para todos.

🌐 Internacionalização

Internacionalização (i18n) vai muito além de traduzir strings. Envolve moedas, fusos horários, formatos de data, compliance regulatório por país (LGPD no Brasil, GDPR na Europa), e até data residency — onde os dados são armazenados fisicamente.

💡 Dimensões da Internacionalização

  • •Localização (l10n) — Tradução de strings, formatos de números e datas, RTL para árabe/hebraico
  • •Moedas e Pagamentos — Multi-currency pricing, tax calculation por região (VAT, ICMS), payment methods locais (PIX, Boleto)
  • •Data Residency — Onde os dados ficam armazenados. GDPR exige dados de europeus na EU. LGPD tem requisitos similares
  • •Timezone Handling — Armazene tudo em UTC, converta na camada de apresentação. Use bibliotecas (date-fns-tz, Luxon)

✓ O que FAZER

  • ✓UTC everywhere no backend, conversão no frontend
  • ✓Usar ICU MessageFormat para pluralização e gênero — não concatene strings
  • ✓Planejar i18n desde o início — retrofitar é 10x mais caro

✗ O que NÃO fazer

  • ✗Hardcodar R$, US$ ou formato de data no código
  • ✗Assumir que Google Translate resolve tradução de produto
  • ✗Armazenar datas em timezone local do servidor

💡 Dica Prática

Use Intl API do JavaScript (Intl.NumberFormat, Intl.DateTimeFormat) para formatação no frontend — é nativa, suporta todos os locales e não precisa de biblioteca externa. No backend, guarde o locale e timezone do tenant como configuração, não do usuário individual.
