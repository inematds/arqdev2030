Pratico
Hands-on

O que e

Tecnicas para validar demanda antes de investir em desenvolvimento: landing pages, fake doors, Wizard of Oz, concierge MVP.

Por que aprender

42% das startups falham por falta de demanda (CB Insights). Ex: O Dropbox validou com um video de 3 minutos — 75k signups em 24h antes de escrever uma linha de codigo. Zappos comecou comprando sapatos na loja e revendendo online para provar demanda.

Conceitos-chave

Fake door testing Wizard of Oz MVP Concierge MVP Landing page + waitlist Pre-selling

O que e

MVP nao e "versao bugada do produto final". E a menor versao que valida a hipotese principal. Uma feature core, bem feita.

Por que aprender

MVPs com escopo grande demais nunca lancam. Ex: O Twitter MVP era so "postar mensagens de 140 caracteres". Sem likes, sem retweets, sem threads, sem trending. Uma feature. Validou que pessoas queriam microblogging.

Conceitos-chave

One-feature MVP Hypothesis-driven development MoSCoW prioritization Definition of "done enough" Time-box (4-6 semanas max)

O que e

Para MVPs, use tecnologia que seu time domina. Nada de experimentar no MVP. Rails, Django, Next.js, Supabase — o que for mais rapido.

Por que aprender

Startups morrem de lentidao, nao de escolha de stack. Ex: Airbnb comecou com Rails. Instagram com Django + PostgreSQL. Figma com C++ no core + React no front. Nenhum usou a "melhor" tecnologia — usaram a que conheciam.

Conceitos-chave

Boring technology thesis Full-stack frameworks BaaS (Supabase, Firebase) No-code/low-code para prototipos Build vs. buy

O que e

Instrumentacao basica desde o primeiro deploy: page views, signups, conversao, retencao, NPS. Nao precisa de data warehouse — Google Analytics + Mixpanel bastam.

Por que aprender

"Ta funcionando" nao e metrica. Ex: Um MVP com 10k signups parece sucesso. Mas se a retencao D7 e 2%, o produto nao resolve o problema. Metrica que importa no MVP: retencao, nao aquisicao.

Conceitos-chave

Pirate metrics (AARRR) North star metric Retention > acquisition Cohort analysis Qualitative feedback

O que e

Ciclos curtos de build-measure-learn. Deploy diario. Feature flags para testar. Feedback loop com usuarios reais.

Por que aprender

A primeira versao nunca esta certa. A velocidade de iteracao e a vantagem competitiva. Ex: O Instagram mudou de Burbn (check-in app com fotos) para Instagram (so fotos) em 8 semanas — perceberam que 80% do uso era nas fotos, pivotaram rapido.

Conceitos-chave

Build-Measure-Learn Weekly sprints Continuous deployment Feature flags User interviews Pivot vs. persevere

O que e

Montar stack 100% free tier: Vercel/Netlify (hosting), Supabase free (DB + auth), Cloudflare (CDN + DNS), GitHub (codigo), n8n self-hosted (automacao).

Por que aprender

Gastar com infra antes de validar e queimar dinheiro. Ex: A INEMA hospeda cursos como HTML estatico via CDN — custo: R$0/mes. Supabase free tier: 500MB de DB, 1GB de storage, 50k MAUs de auth. Mais que suficiente para validar.

Conceitos-chave

Free tier stacking Static-first architecture Serverless free tiers Cost monitoring When to upgrade

O que e

Sinais de que o MVP validou e e hora de investir: retencao consistente, willingness to pay, feature requests repetidos, product-market fit score >40%.

Por que aprender

Investir antes do fit e desperdicio. Depois do fit, cada dia sem investir e oportunidade perdida. Ex: Sean Ellis test — "Como voce se sentiria se nao pudesse mais usar este produto?" Se >40% dizem "muito desapontado", voce tem product-market fit. Hora de escalar.

Conceitos-chave

Product-market fit signals Sean Ellis test Retention curves Revenue validation Tech debt paydown plan

O que e

Boas praticas: nomes no plural (/users, /orders), HTTP verbs corretos, status codes semanticos, paginacao, filtros, sorting.

Por que aprender

API mal desenhada gera duvidas infinitas e integracoes frageis. Ex: Stripe e referencia em API design — /v1/charges, /v1/customers, POST para criar, GET para listar. Tao intuitiva que devs integram sem ler docs.

Conceitos-chave

Resource naming HTTP methods Status codes (2xx, 4xx, 5xx) Pagination (cursor vs. offset) Filtering HATEOAS

O que e

Autenticacao (identidade) via JWT, OAuth2, API keys. Autorizacao (permissoes) via RBAC, ABAC, scopes.

Por que aprender

Auth errada e a brecha #1 em APIs. Ex: Em 2019, a API do Facebook tinha bug de autorizacao — permitia buscar tokens de amigos de amigos. 50M de contas expostas. Multa de US$5B da FTC.

Conceitos-chave

JWT (access + refresh tokens) OAuth2 flows API keys vs. tokens RBAC Rate limiting per key Token expiration

O que e

OpenAPI (Swagger) e o padrao para documentar APIs REST. Schema define endpoints, parametros, responses. Gera docs interativas e SDKs automaticamente.

Por que aprender

API sem docs e API sem usuarios. Ex: A Twilio gasta mais em docs que em marketing. Resultado: devs integram em minutos, conversao 10x maior que APIs com docs ruins.

Conceitos-chave

OpenAPI 3.0 spec Swagger UI Postman collections Code-first vs. spec-first SDK generation

O que e

Estrategias para evoluir APIs sem breaking changes: URL versioning (/v1/, /v2/), header versioning, additive changes, deprecation policy.

Por que aprender

Breaking change em API publica = clientes furiosos e integracoes quebradas. Ex: O Twitter v1 → v2 foi um desastre — mudaram tudo de uma vez, quebraram milhares de apps. Stripe faz diferente: versionamento por data (2023-10-16), backward compatible, deprecation com 12 meses de aviso.

Conceitos-chave

URL vs. header vs. query param versioning Additive-only changes Deprecation policy Migration guides API changelog

O que e

Padroes para lidar com falhas em APIs externas: retry com exponential backoff, circuit breaker, timeout, fallback, bulkhead.

Por que aprender

Se seu sistema depende de API que cai, seu sistema cai junto. Ex: O Netflix tem o Hystrix (circuit breaker) — quando a API de recomendacao cai, mostra "Top 10 do Brasil" (fallback estatico) em vez de erro. Usuario nem percebe.

Conceitos-chave

Circuit breaker (open/half-open/closed) Exponential backoff Timeout budget Fallback strategies Bulkhead pattern

O que e

Como implementar webhooks (producer): endpoint publico, retry policy, signature (HMAC), logging. Como consumir (consumer): validacao, idempotencia, processamento async.

Por que aprender

Webhooks sao como APIs externas chamam VOCE. Ex: Stripe envia webhook quando pagamento confirma. Seu n8n recebe, valida assinatura HMAC, processa idempotentemente (nao libera acesso 2x se o webhook chega 2x).

Conceitos-chave

HMAC signature verification Idempotency keys Async processing Retry handling Dead letter queue n8n as webhook receiver

O que e

Proxy reverso que centraliza: autenticacao, rate limiting, caching, logging, transformacao de requests. Kong, AWS API Gateway, Traefik.

Por que aprender

Sem gateway, cada servico reimplementa cross-cutting concerns. Ex: Com API Gateway: auth uma vez, rate limit centralizado, logging unificado. Sem: cada microsservico tem sua implementacao (bugada e inconsistente) de auth e logging.

Conceitos-chave

Reverse proxy Authentication offloading Rate limiting Request/response transformation Caching Analytics

O que e

Banco relacional open-source com extensoes para JSON (jsonb), full-text search, geoespacial (PostGIS), vetores (pgvector). E o "canivete suico" dos bancos.

Por que aprender

PostgreSQL cobre 90% dos casos de uso sem precisar de outro banco. Ex: O Instagram usou PostgreSQL puro (sem NoSQL) para escalar a 1B de fotos. O Supabase e PostgreSQL com API REST automatica. pgvector transforma em vector database para IA.

Conceitos-chave

ACID transactions jsonb para dados semi-estruturados Indexes (B-tree, GIN, GiST) Full-text search pgvector para embeddings Extensions

O que e

Normalizacao evita redundancia (3NF). Desnormalizacao otimiza leitura. Na pratica, use normalizacao por padrao e desnormalize seletivamente para performance.

Por que aprender

Modelo errado gera queries lentas e bugs de consistencia. Ex: Tabela "pedidos" com endereco do cliente inline (desnormalizado) — quando o cliente muda de endereco, pedidos antigos mostram o novo endereco. Bug classico de desnormalizacao sem snapshots.

Conceitos-chave

1NF/2NF/3NF When to denormalize Materialized views Database views Query optimization

O que e

Versionamento de schema com ferramentas de migration (Alembic, Flyway, Prisma Migrate). Cada mudanca e um script versionado que roda em sequencia.

Por que aprender

ALTER TABLE em producao sem migration = desastre. Ex: Adicionar coluna NOT NULL sem default em tabela com 100M de rows trava o banco por horas. Migration correta: (1) add column nullable, (2) backfill, (3) add NOT NULL constraint.

Conceitos-chave

Migration tools Backward-compatible migrations Zero-downtime migrations Rollback strategies Data backfill

O que e

Bancos nao-relacionais para casos especificos: MongoDB (documentos), Redis (cache/sessoes), DynamoDB (key-value escalavel), Neo4j (grafos).

Por que aprender

Usar NoSQL onde relacional basta e over-engineering. Usar relacional onde NoSQL brilha e tortura. Ex: Grafo social (quem segue quem) em PostgreSQL exige JOINs recursivos caros. Em Neo4j, e uma query de 1 linha: MATCH (a)-[:FOLLOWS*1..3]->(b) RETURN b.

Conceitos-chave

Document stores Key-value stores Column-family Graph databases Polyglot persistence When NOT to use NoSQL

O que e

Armazenar resultados frequentes em memoria (Redis, Memcached) ou CDN. Reduz carga no banco e latencia para o usuario.

Por que aprender

Cache bem implementado pode reduzir carga no banco em 90%+. Ex: O Twitter armazena timelines em Redis — sem cache, cada refresh geraria queries em centenas de tabelas. Com cache, serve de memoria em <5ms.

Conceitos-chave

Cache-aside pattern Write-through vs. write-behind Cache invalidation TTL strategies Redis data structures CDN caching

O que e

Indices aceleram leitura ao custo de escrita mais lenta e mais espaco. B-tree (default), Hash (igualdade), GIN (full-text/jsonb), GiST (geoespacial).

Por que aprender

Query sem indice adequado = full table scan = timeout. Ex: Tabela com 10M de rows. SELECT * WHERE email = 'x' sem indice: 15 segundos. Com indice B-tree em email: 0.5ms. 30.000x mais rapido.

Conceitos-chave

B-tree Hash GIN GiST Composite indexes EXPLAIN ANALYZE Index bloat Partial indexes Covering indexes

O que e

Plataforma BaaS open-source baseada em PostgreSQL. API REST automatica (PostgREST), auth built-in, storage, realtime subscriptions, edge functions.

Por que aprender

Substitui semanas de setup por minutos. Ex: CRUD completo com auth: criar projeto no Supabase → definir tabelas → Row Level Security → frontend conecta via SDK. Zero backend code para operacoes padrao.

Conceitos-chave

PostgREST auto-API Row Level Security (RLS) Supabase Auth Storage com policies Realtime subscriptions Edge Functions

O que e

Paginas HTML puras servidas via CDN. Zero JavaScript obrigatorio. Carrega em <1s em qualquer conexao. Custo zero de hosting.

Por que aprender

SPA e overkill para 80% dos sites. Ex: A documentacao do Tailwind CSS e HTML estatico gerado — serve milhoes de page views/mes via CDN por centavos. A INEMA serve cursos como HTML estatico: zero bundle, zero build, zero framework.

Conceitos-chave

Static site generators (Astro, Hugo, 11ty) CDN deployment Progressive enhancement Core Web Vitals Tailwind CSS

O que e

Single Page Applications para UIs com muita interatividade: dashboards, editores, apps real-time. React/Next.js, Vue/Nuxt, Svelte/SvelteKit.

Por que aprender

SPA adiciona complexidade (build, bundle, hydration, state management). So vale quando a interatividade exige. Ex: Figma PRECISA ser SPA — e um editor visual com colaboracao real-time. Um blog nao precisa.

Conceitos-chave

SPA vs. MPA SSR vs. CSR vs. SSG React/Next.js Vue/Nuxt Svelte/SvelteKit Hydration cost

O que e

Biblioteca de componentes padronizados: botoes, forms, modais, cards. Compartilhados entre projetos. Documentados com Storybook.

Por que aprender

Sem design system, cada tela reinventa o botao. Ex: O Material UI do Google tem 80+ componentes. Resultado: qualquer app Google tem look consistente. Qualquer dev novo produz UI consistente no dia 1.

Conceitos-chave

Component library Storybook Design tokens Atomic design Tailwind + Headless UI shadcn/ui

O que e

Layouts que se adaptam a qualquer tela: mobile, tablet, desktop. Mobile-first: design para mobile, depois expande para desktop.

Por que aprender

60%+ do trafego web e mobile. Site que nao funciona no celular perde maioria dos usuarios. Ex: O Google penaliza sites nao-responsivos no ranking de busca. Core Web Vitals incluem mobile performance como fator de ranking.

Conceitos-chave

Mobile-first CSS Breakpoints Flexbox/Grid Touch targets Viewport meta tag Tailwind responsive prefixes

O que e

Web Content Accessibility Guidelines (WCAG): contraste, navegacao por teclado, screen readers, alt text, ARIA labels.

Por que aprender

15% da populacao tem alguma deficiencia. Acessibilidade e requisito legal em muitos paises. Ex: Domino's Pizza perdeu processo de US$10M por site nao-acessivel a deficientes visuais. Em 2024, 97% dos top 1M sites tem erros de acessibilidade (WebAIM).

Conceitos-chave

WCAG 2.1 AA Semantic HTML ARIA roles Keyboard navigation Color contrast Screen reader testing

O que e

Metricas do Google: LCP (Largest Contentful Paint), FID (First Input Delay), CLS (Cumulative Layout Shift). Afetam SEO e experiencia.

Por que aprender

53% dos usuarios mobile abandonam se o site leva >3s para carregar (Google). Ex: O Pinterest reduziu loading time em 40% e viu aumento de 15% no signup e 44% na receita de ads. Performance = dinheiro.

Conceitos-chave

LCP FID/INP CLS Image optimization (WebP, lazy loading) Code splitting Tree shaking Font loading strategy

O que e

Suporte a dark/light mode com toggle e deteccao automatica de preferencia do sistema. CSS variables ou Tailwind dark: prefix.

Por que aprender

Dark mode reduz fadiga visual e economiza bateria em OLED. 82% dos usuarios mobile usam dark mode (Android Authority, 2023). Ex: Nosso proprio curso usa dark mode como padrao com toggle — Tailwind dark: prefix + class strategy + localStorage.

Conceitos-chave

prefers-color-scheme Tailwind dark mode CSS custom properties Theme toggle with localStorage System preference detection

O que e

Distribuicao ideal: 70% unitarios (rapidos, baratos), 20% integracao (medios), 10% E2E (lentos, caros). Base larga, topo estreito.

Por que aprender

Piramide invertida (muitos E2E, poucos unitarios) = CI lento e flaky. Ex: O Google tem 80% de testes unitarios, 15% de integracao, 5% E2E. Suite roda em <5 minutos. Spotify tinha piramide invertida — CI levava 45 minutos e falhava 20% das vezes por flaky tests.

Conceitos-chave

Unit tests Integration tests E2E tests Test doubles (mock/stub/spy) Test coverage (80% rule) Flaky tests

O que e

Ciclo Red-Green-Refactor: escreve teste que falha → escreve codigo minimo para passar → refatora. O teste guia o design.

Por que aprender

TDD produz codigo mais testavel e com menos bugs. Ex: Kent Beck (criador do TDD) relata que equipes com TDD tem 40-80% menos bugs em producao (IBM Research). O custo e 15-35% mais tempo de desenvolvimento — que se paga em manutencao.

Conceitos-chave

Red-Green-Refactor Test-first design When TDD helps (business logic) When TDD hurts (UI, exploratory) Outside-in vs. inside-out

O que e

Testa interacao entre componentes: API + banco, servico A + servico B. Usa banco real (nao mock), APIs reais ou containers.

Por que aprender

Mocks podem mascarar problemas reais. Ex: Testes unitarios com mock do banco passaram, mas em producao a migration nao rodou e a coluna nao existia. Teste de integracao com banco real teria pego.

Conceitos-chave

Testcontainers Database fixtures API contract testing Integration test isolation Docker Compose for testing

O que e

Automacao de browser que simula interacao real: login, navegacao, formularios, checkout. Playwright, Cypress, Selenium.

Por que aprender

E a ultima linha de defesa antes do usuario. Ex: O Playwright testa o checkout completo: adiciona produto → preenche dados → confirma pagamento → verifica confirmacao. Se qualquer step quebra, o teste falha antes do deploy.

Conceitos-chave

Playwright (recomendado) Cypress Selenium Page Object Pattern Visual regression testing CI/CD integration

O que e

Verifica que a API respeita seu contrato (schema, tipos, formato). Consumer-driven: o consumidor define o que espera, o producer garante.

Por que aprender

Em microsservicos, mudanca no producer quebra o consumer. Contract tests pegam isso antes do deploy. Ex: O Pact framework: frontend define "espero GET /users retorne [{id: number, name: string}]". Se o backend muda o tipo de id para string, o teste falha na CI do backend.

Conceitos-chave

Pact Consumer-driven contracts Provider verification Schema validation Breaking change detection

O que e

Pipeline automatizado: commit → lint → unit tests → integration tests → E2E → deploy. GitHub Actions, GitLab CI, CircleCI.

Por que aprender

Testes que nao rodam automaticamente sao ignorados. Ex: GitHub Actions workflow: push → roda 500 unit tests em 30s → 50 integration tests em 2min → 10 E2E em 3min → deploy. Se qualquer step falha, PR nao merges.

Conceitos-chave

GitHub Actions Test parallelization Caching Flaky test detection Test reporting Quality gates

O que e

Coverage mostra quanto codigo e testado. Mutation testing mostra se os testes realmente validam. Metricas de qualidade guiam investimento em testes.

Por que aprender

100% coverage com assertions fracas e ilusao de seguranca. Ex: Mutation testing (Stryker, mutmut) modifica o codigo e verifica se testes pegam a mutacao. Se trocar > por >= e nenhum teste falha, o teste e fraco. Meta: mutation score >80%.

Conceitos-chave

Code coverage (line, branch, path) Mutation testing Cyclomatic complexity Technical debt ratio SonarQube
