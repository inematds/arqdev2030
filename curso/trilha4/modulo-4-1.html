MÓDULO 4.1

🚀 Do Zero ao MVP

Construir um MVP não é construir um produto ruim — é construir o produto certo com o mínimo de esforço. Neste módulo, você vai aprender a validar antes de codar, escolher escopo, montar a stack ideal e iterar rapidamente até encontrar product-market fit.

1

🔍 Validação antes do Código

O maior erro de desenvolvedores é começar a codar antes de validar a ideia. A maioria dos MVPs falha não por problemas técnicos, mas porque resolvem um problema que ninguém tem. Antes de escrever uma linha de código, você precisa ter evidências de que alguém pagaria pelo que você quer construir.

💡 Técnicas de Validação Pré-Código

  • •Landing Page Test — Crie uma página descrevendo o produto e colete e-mails de interessados antes de construir
  • •Entrevistas com usuários — Converse com 10-20 potenciais usuários. Pergunte sobre o problema, não sobre a solução
  • •Concierge MVP — Faça manualmente o que o software faria. Se funcionar, automatize
  • •Wizard of Oz — Interface funcional por fora, humano operando por dentro

📊 Caso Real: Dropbox

Drew Houston validou o Dropbox com um vídeo de 3 minutos mostrando como o produto funcionaria — antes de escrever uma linha de código:

  • Antes do vídeo: 5.000 inscritos na lista de espera
  • Depois do vídeo: 75.000 inscritos em 24 horas
  • Lição: Validação não exige produto pronto, exige clareza sobre o problema

💡 Dica Prática

Use a regra do “Mom Test” nas entrevistas: nunca pergunte “você usaria isso?” (todos dizem sim). Pergunte “como você resolve esse problema hoje?” e “quanto tempo/dinheiro você gasta com isso?”. Se a pessoa não tem o problema, não vai pagar pela solução.

2

🎯 Escolhendo o Escopo

O escopo do MVP deve ser brutalmente reduzido. Não é a versão 1.0 do produto — é o experimento mínimo para validar a hipótese central. Se o seu MVP leva mais de 4-6 semanas para construir, o escopo está grande demais.

💡 Framework de Priorização

Use a matriz de impacto vs. esforço para decidir o que entra no MVP:

  • •Alto impacto + Baixo esforço — Faça primeiro (quick wins)
  • •Alto impacto + Alto esforço — Planeje para a v2
  • •Baixo impacto + Baixo esforço — Só se sobrar tempo
  • •Baixo impacto + Alto esforço — Elimine sem dó

✓ O que FAZER

  • ✓Definir uma única métrica de sucesso para o MVP
  • ✓Cortar funcionalidades até doer — menos é mais
  • ✓Usar user stories para mapear o fluxo crítico

✗ O que NÃO fazer

  • ✗Construir sistema de permissões complexo no dia 1
  • ✗Adicionar internacionalização antes de ter 10 usuários
  • ✗Gastar semanas no design perfeito em vez de validar
3

🛠️ Stack para MVP

A melhor stack para MVP é a que você já domina. Não é hora de aprender Rust, Elixir ou a última framework da moda. Velocidade de execução é tudo. Frameworks full-stack, BaaS e plataformas low-code são seus aliados.

💡 Stacks Recomendadas para MVP em 2026

  • •Next.js + Supabase — Full-stack React com banco, auth e storage prontos. Deploy na Vercel em minutos
  • •Remix + PlanetScale — SSR moderno com MySQL serverless. Ótimo para SEO
  • •Rails + Fly.io — Produtividade máxima, convention over configuration. Deploy global
  • •Django + Railway — Ecossistema Python, admin pronto, ORM poderoso

⚠️ Armadilha: Resume-Driven Development

Se você está escolhendo uma tecnologia porque quer colocar no currículo e não porque resolve o problema, você está priorizando sua carreira sobre o produto. No MVP, pragmatismo vence elegância. Use o que é chato, testado e documentado.

💡 Dica Prática

Antes de escolher a stack, responda: “Consigo fazer deploy sozinho em menos de 1 hora?” Se a resposta for não, a stack é complexa demais para um MVP. Procure soluções com deploy integrado (Vercel, Railway, Fly.io, Render).

4

📊 Métricas desde o Dia 1

Se você não mede, não sabe se está funcionando. Métricas não são luxo de empresa grande — são a bússola do MVP. Sem dados, você toma decisões baseadas em achismo. Com dados, você sabe exatamente o que iterar.

💡 Métricas Pirata (AARRR)

O framework de Dave McClure para medir o que importa:

  • •Acquisition — Como os usuários chegam? (visitas, sources)
  • •Activation — Eles têm uma boa primeira experiência? (signup, onboarding completo)
  • •Retention — Eles voltam? (DAU/MAU, sessões por semana)
  • •Revenue — Eles pagam? (conversão, ARPU)
  • •Referral — Eles indicam? (NPS, convites enviados)

📊 Ferramentas Gratuitas para Começar

  • PostHog — Analytics open-source com feature flags, session replay e funnels
  • Plausible — Analytics leve e privacy-friendly (alternativa ao Google Analytics)
  • Sentry — Monitoramento de erros com tier gratuito generoso
  • Hotjar — Heatmaps e gravação de sessões para entender comportamento

💡 Dica Prática

No MVP, foque em uma única métrica que importa (OMTM). Se for um SaaS, provavelmente é retenção na semana 1. Se for e-commerce, é taxa de conversão. Terá muitas métricas interessantes — ignore todas menos a principal até validá-la.

5

🔄 Iteração Rápida

O ciclo Build → Measure → Learn do Lean Startup é a base da iteração rápida. O objetivo não é construir rápido por construir — é aprender rápido. Cada iteração deve responder uma pergunta específica sobre o produto.

🛠️ Ciclo de Iteração em 4 Passos

1

Formule a hipótese

“Acreditamos que [mudança X] vai causar [resultado Y], e saberemos que funcionou quando [métrica Z] atingir [valor W].”

2

Construa o mínimo

Implemente apenas o necessário para testar a hipótese. Nada mais. Feature flags ajudam a lançar parcialmente.

3

Meça os resultados

Colete dados quantitativos (métricas) e qualitativos (feedback direto). Não espere semanas — dias são suficientes.

4

Decida: pivotar ou perseverar

Se a hipótese foi validada, dobre a aposta. Se não, pivote rápido. Não existe fracasso — existe aprendizado.

💡 Dica Prática

Defina sprints de 1 semana no início do MVP. Sprints de 2 semanas são para times maiores. Com 1-3 pessoas, 1 semana é suficiente para construir, medir e aprender. Se uma feature não cabe em 1 sprint, quebre-a em partes menores.

6

💰 Custo Zero de Infra

Em 2026, é possível lançar um MVP completo sem gastar um centavo com infraestrutura. Free tiers generosos, plataformas serverless e serviços gerenciados permitem começar sem investimento em infra. Gaste seu dinheiro com validação, não com servidores.

💡 Stack de Custo Zero

  • •Hosting: Vercel/Netlify (free tier), Cloudflare Pages (ilimitado)
  • •Banco: Supabase (500MB free), Neon (512MB free), Turso (9GB free)
  • •Auth: Supabase Auth, Clerk (10k MAU free), Auth.js (open-source)
  • •E-mail: Resend (100 emails/dia free), Loops (free tier)
  • •Storage: Cloudflare R2 (10GB free, sem egress), Supabase Storage
  • •Analytics: PostHog (1M eventos/mês free), Plausible (self-host)

✓ O que FAZER

  • ✓Usar serviços gerenciados em vez de hospedar tudo
  • ✓Planejar migração quando escalar (saber os limites)
  • ✓Monitorar consumo para evitar surpresas

✗ O que NÃO fazer

  • ✗Configurar Kubernetes para servir 50 usuários
  • ✗Pagar por AWS/GCP enterprise antes de faturar
  • ✗Ficar preso a vendor lock-in sem plano de saída
7

🏆 Do MVP ao Produto

O MVP validou a hipótese. E agora? A transição de MVP para produto é o momento mais perigoso. Muitos founders cometem o erro de continuar no modo MVP para sempre (acumulando dívida técnica) ou de reescrever tudo do zero (perdendo meses).

💡 Sinais de que É Hora de Evoluir

  • •Retenção consistente — Usuários voltam semana após semana
  • •Receita recorrente — Pessoas pagam (ou pagariam) pelo produto
  • •Bugs frequentes — A dívida técnica começa a impactar a experiência
  • •Demanda por features — Usuários pedem funcionalidades que exigem refatoração

📊 Caso Real: Notion

O Notion reescreveu o produto do zero duas vezes antes de encontrar o product-market fit:

  • V1 (2013): Muito complexo, poucos usuários entendiam
  • V2 (2015): Simplificado, mas sem diferença clara dos concorrentes
  • V3 (2018): Blocos como primitivo universal — explosão de crescimento
  • Lição: Às vezes a reescrita é necessária, mas só quando você entende o porquê da versão anterior ter falhado

💡 Dica Prática

Use a estratégia do “Strangler Fig Pattern” para evoluir o MVP: em vez de reescrever tudo, substitua componentes gradualmente. Extraia o código mais crítico primeiro, adicione testes, e evolua incrementalmente. O MVP vira produto sem parar de funcionar.

📋 Resumo do Módulo

✓
Valide antes de codar — Landing pages, entrevistas e concierge MVP custam zero e evitam meses de trabalho desperdiçado
✓
Escopo brutal — Se leva mais de 6 semanas, está grande demais. Corte sem dó
✓
Stack que você domina — Velocidade > elegância. Use tecnologias maduras e plataformas com deploy fácil
✓
Meça tudo — Métricas AARRR + OMTM desde o dia 1. Sem dados, você está no escuro
✓
Itere em sprints curtos — Build → Measure → Learn em ciclos de 1 semana
✓
Infra de custo zero — Free tiers generosos permitem lançar sem investimento inicial
✓
MVP ≠ produto final — Evolua gradualmente com Strangler Fig Pattern quando tiver product-market fit
