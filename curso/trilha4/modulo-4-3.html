🐘 PostgreSQL

PostgreSQL é o banco de dados mais versátil do mundo. Relacional com suporte a JSON, full-text search, geoespacial, vetores (pgvector para IA) e extensões infinitas. Se você não sabe qual banco usar, a resposta é PostgreSQL. Ponto.

💡 Por que PostgreSQL em 2026

  • •JSONB nativo — Armazene dados semi-estruturados sem precisar de MongoDB
  • •pgvector — Busca vetorial para embeddings de IA direto no banco
  • •Full-text search — Busca textual poderosa sem precisar de Elasticsearch
  • •Row Level Security (RLS) — Autorização no nível do banco, usado pelo Supabase
  • •Ecossistema — Neon (serverless), Supabase (BaaS), Crunchy (enterprise), Tembo (extensible)

📊 Quem Usa PostgreSQL

  • Instagram — Bilhões de rows, PostgreSQL como banco primário desde o dia 1
  • Discord — Trilhões de mensagens armazenadas com sharding sobre PostgreSQL
  • Supabase — Todo o produto é construído em cima de PostgreSQL com extensões
  • Apple, Spotify, Reddit — Todos usam PostgreSQL em sistemas críticos

💡 Dica Prática

Para desenvolvimento local, use Docker: docker run -e POSTGRES_PASSWORD=dev -p 5432:5432 postgres:16. Para produção, use Neon (serverless, free tier generoso) ou Supabase (PostgreSQL + auth + API automática).

📐 Modelagem Relacional

Modelagem relacional é a base de tudo. Entender normalização, relacionamentos, chaves e constraints é o que separa um banco bem estruturado de um pesadelo de manutenção. Erros de modelagem são os mais caros de corrigir depois.

💡 Princípios de Modelagem

  • •Normalização (3NF) — Elimine redundância. Cada dado em um único lugar. Mas não exagere
  • •Foreign Keys — Sempre use constraints de integridade referencial. O banco é a última linha de defesa
  • •UUIDs vs. Auto-increment — UUIDs são melhores para sistemas distribuídos. Auto-increment para simplicidade
  • •Timestamps — Sempre tenha created_at e updated_at em toda tabela
  • •Soft delete — Use deleted_at em vez de deletar registros em domínios críticos

✓ Boas Práticas

  • ✓Nomes de tabela no plural e snake_case
  • ✓Usar ENUMs do banco para estados finitos
  • ✓Documentar o schema com COMMENT ON

✗ Antipadrões

  • ✗Tabela “God Table” com 50+ colunas
  • ✗Armazenar JSON em VARCHAR (use JSONB)
  • ✗Ignorar foreign keys “por performance”

🔃 Migrations

Migrations são o controle de versão do banco de dados. Assim como Git rastreia mudanças no código, migrations rastreiam mudanças no schema. Sem elas, é impossível manter consistência entre ambientes (dev, staging, produção).

💡 Boas Práticas de Migrations

  • •Uma migration por mudança — Pequenas, atmômicas, reversíveis
  • •Sempre teste rollback — Se você não pode reverter, não faça deploy na sexta
  • •Sem dados em migrations — Migrations mudam schema, seeds inserem dados
  • •Nomes descritivos — 20260321_add_status_to_orders, não migration_42

📊 Ferramentas de Migration

  • Prisma Migrate — Gera migrations a partir do schema declarativo. TypeScript-first
  • Drizzle Kit — Leve, rápido, migrations SQL puras com type safety
  • Knex.js — Query builder com sistema de migrations maduro
  • Flyway / Liquibase — Enterprise, Java-based, robusto para projetos grandes
  • Supabase CLI — Migrations integradas com diff automático do schema

⚠️ Migrations Perigosas em Produção

Estas operações podem derrubar sua aplicação se executadas descuidadamente:

  • ✗ALTER TABLE ... ADD COLUMN ... NOT NULL sem default — Bloqueia a tabela inteira
  • ✗CREATE INDEX sem CONCURRENTLY — Trava writes durante a criação
  • ✗DROP COLUMN de uma coluna ainda usada pelo código em produção

📃 NoSQL

NoSQL não é “melhor que SQL” — é diferente, para problemas diferentes. Bancos NoSQL brilham quando o modelo relacional não encaixa: dados hierárquicos, alta velocidade de escrita, schemas flexíveis ou escala horizontal massiva.

💡 Tipos de NoSQL e Quando Usar

  • •Documento (MongoDB, Firestore) — Dados hierárquicos, schemas variáveis. Bom para CMS, catálogos
  • •Key-Value (Redis, DynamoDB) — Latência ultra-baixa, cache, sessões, rate limiting
  • •Wide-Column (Cassandra, ScyllaDB) — Escrita massiva, séries temporais, IoT
  • •Grafo (Neo4j, DGraph) — Relacionamentos complexos: redes sociais, recomendações, fraude

💡 Dica Prática

Em 90% dos casos, PostgreSQL com JSONB resolve o que você usaria MongoDB. Só considere NoSQL dedicado quando: (1) o volume de dados excede o que PostgreSQL aguenta, (2) o padrão de acesso é fundamentalmente diferente (grafos, time-series), ou (3) você precisa de latência sub-millisecond (Redis).

⚡ Caching

“There are only two hard things in Computer Science: cache invalidation and naming things.” Caching é a forma mais eficaz de melhorar performance, mas também a fonte mais comum de bugs sutis. Entender quando e como cachear é uma habilidade crítica.

💡 Camadas de Cache

  • •Browser cache — Cache-Control headers, ETags, Service Workers
  • •CDN cache — Cloudflare, Vercel Edge, CloudFront. Cache na borda, perto do usuário
  • •Application cache — In-memory (Node.js Map), Redis, Memcached
  • •Database cache — Query cache, materialized views, connection pooling (PgBouncer)

🛠️ Estratégias de Invalidação

1

TTL (Time-To-Live)

Cache expira automaticamente após X segundos. Simples, mas pode servir dados desatualizados.

2

Write-through

Atualiza cache e banco simultaneamente. Consistência forte, mas mais lento na escrita.

3

Cache-aside (Lazy loading)

Leia do cache; se miss, leia do banco e popule o cache. Padrão mais comum e flexível.

4

Event-driven invalidation

Invalide o cache quando um evento de mudança ocorre. Mais complexo, mais preciso.

📊 Índices e Performance

Índices são a diferença entre uma query que leva 2ms e uma que leva 20 segundos. Mas índices demais pioram a escrita e consomem espaço. O segredo é indexar com base nos padrões reais de consulta, não em suposicões.

💡 Tipos de Índice no PostgreSQL

  • •B-tree (padrão) — Para igualdade e range queries. Cobre 90% dos casos
  • •GIN — Para JSONB, arrays, full-text search. Essencial para dados semi-estruturados
  • •GiST — Para dados geoespaciais (PostGIS) e range types
  • •BRIN — Para tabelas muito grandes com dados naturalmente ordenados (timestamps)
  • •Partial index — WHERE status = 'active' — Indexa só o que importa

✓ O que FAZER

  • ✓Usar EXPLAIN ANALYZE antes de otimizar
  • ✓Indexar colunas usadas em WHERE e JOIN
  • ✓Usar composite indexes para queries multi-coluna

✗ O que NÃO fazer

  • ✗Indexar todas as colunas “por via das dúvidas”
  • ✗Otimizar queries que rodam 1x por dia
  • ✗Usar SELECT * quando só precisa de 3 colunas

💡 Dica Prática

Instale a extensão pg_stat_statements em produção. Ela mostra as queries mais lentas e mais executadas. Otimize as top 5 queries por tempo total (frequência × duração) — isso cobre 80% dos ganhos de performance.

⚡ Supabase na Prática

Supabase é o “Firebase open-source” construído sobre PostgreSQL. Oferece banco, autenticação, storage, realtime e edge functions em uma plataforma integrada. Para MVPs e startups, é a forma mais rápida de ter um backend completo sem construir do zero.

💡 O que o Supabase Oferece

  • •PostgreSQL completo — Acesso direto ao banco, sem abstrações que limitam
  • •Auto-generated API — PostgREST gera API REST automaticamente a partir das tabelas
  • •Realtime — WebSockets para mudanças no banco em tempo real
  • •Auth integrado — Email, OAuth, magic links, MFA — tudo pronto
  • •Edge Functions — Deno-based serverless functions para lógica customizada
  • •Row Level Security — Autorização no nível do banco — seguro por default

📊 Supabase em Números

  • 1M+ projetos criados na plataforma
  • Open-source — Você pode self-host com Docker Compose
  • Free tier — 500MB de banco, 1GB de storage, 50k MAU de auth
  • Usado por: Mozilla, Peerlist, HappyScribe, Chatbase

✓ O que FAZER

  • ✓Usar RLS para segurança no nível do banco
  • ✓Usar Supabase CLI para migrations e desenvolvimento local
  • ✓Combinar com Next.js ou Remix para full-stack completo

✗ O que NÃO fazer

  • ✗Expor o service_role key no client-side
  • ✗Ignorar RLS e confiar apenas no frontend
  • ✗Usar Supabase como único backend para lógica de negócio complexa

💡 Dica Prática

Comece com Supabase para MVP e migre para PostgreSQL gerenciado (Neon, RDS) quando precisar de mais controle. Como é PostgreSQL puro por baixo, a migração é trivial: exporte com pg_dump e importe com pg_restore. Zero vendor lock-in.
