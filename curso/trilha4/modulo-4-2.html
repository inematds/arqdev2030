🛠️ Design de APIs REST

Uma API bem projetada é previsível, consistente e auto-descritiva. Desenvolvedores que consomem sua API devem entender o que cada endpoint faz sem ler documentação. REST não é um protocolo — é um estilo arquitetural com convenções que, quando seguidas, tornam a API intuitiva.

💡 Princípios de Design REST

  • •Recursos como substantivos — /users, /orders, nunca /getUsers
  • •Verbos HTTP corretos — GET (ler), POST (criar), PUT/PATCH (atualizar), DELETE (remover)
  • •Status codes semnânticos — 200 OK, 201 Created, 400 Bad Request, 404 Not Found, 422 Unprocessable
  • •Paginação e filtragem — ?page=2&limit=20&status=active
  • •Respostas consistentes — Sempre o mesmo formato: { data, meta, errors }

✓ Bom Design

  • ✓GET /api/v1/users/123/orders
  • ✓POST /api/v1/orders
  • ✓PATCH /api/v1/orders/456

✗ Design Ruim

  • ✗GET /api/getUserOrders?id=123
  • ✗POST /api/createNewOrder
  • ✗POST /api/updateOrder/456

💡 Dica Prática

Antes de implementar, escreva a documentação da API primeiro (API-first design). Isso força você a pensar na interface antes da implementação. Ferramentas como Swagger Editor permitem projetar a API visualmente.

🔒 Autenticação e Autorização

Autenticação responde “quem é você?”. Autorização responde “o que você pode fazer?”. Confundir os dois é um dos erros de segurança mais comuns. Em 2026, nunca implemente auth do zero — use serviços especializados.

💡 Padrões de Autenticação

  • •JWT (JSON Web Tokens) — Stateless, escalável, mas não revogável. Ideal para APIs entre serviços
  • •OAuth 2.0 — Delegação de acesso. “Login com Google/GitHub” usa OAuth
  • •API Keys — Simples, bom para serviços M2M. Não use para usuários finais
  • •Session Cookies — Stateful, mais seguro contra XSS. Ideal para aplicações web tradicionais

⚠️ Nunca Faça Isso

  • ✗Armazenar senhas em texto puro — Use bcrypt ou Argon2
  • ✗Guardar JWT no localStorage — Vulnerável a XSS. Use httpOnly cookies
  • ✗Implementar crypto própria — Use bibliotecas auditadas e battle-tested
  • ✗Confiar no client-side para autorização — Sempre valide no servidor

💡 Dica Prática

Use Supabase Auth, Clerk ou Auth.js para autenticação. Para autorização complexa (RBAC, ABAC), considere CASL ou Cerbos. A regra de ouro: nunca gaste tempo implementando auth quando existem soluções prontas e auditadas.

📝 Documentação com OpenAPI

API sem documentação é API que ninguém usa. OpenAPI (antigo Swagger) é o padrão da indústria para descrever APIs REST de forma estruturada. A partir de um arquivo YAML/JSON, você gera documentação interativa, SDKs e testes automaticamente.

💡 Benefícios do OpenAPI

  • •Documentação interativa — Swagger UI permite testar endpoints direto no browser
  • •Geração de código — SDKs para TypeScript, Python, Go gerados automaticamente
  • •Validação automática — Middleware valida request/response contra o schema
  • •Contrato entre times — Frontend e backend concordam na interface antes de implementar

📊 Ferramentas de Documentação

  • Swagger UI — Interface clássica, testável, integrada com a maioria dos frameworks
  • Redoc — Documentação bonita e responsiva, ideal para APIs públicas
  • Scalar — Alternativa moderna ao Swagger UI, com UX superior
  • Bruno / Insomnia — Clientes HTTP que importam OpenAPI para testes manuais

🔄 Versionamento

APIs evoluem. Clientes não devem quebrar quando você muda a API. Versionamento é a estratégia para introduzir mudanças sem destruir integrações existentes. A escolha da estratégia impacta diretamente a complexidade de manutenção.

💡 Estratégias de Versionamento

  • •URL Path — /api/v1/users → Mais comum, fácil de entender e rotear
  • •Header — Accept: application/vnd.api+json;version=2 → Mais “RESTful”, menos visível
  • •Query Parameter — ?version=2 → Simples, mas polui a URL
  • •Sem versionamento — Evoluir aditivamente (nunca remover campos, só adicionar). Stripe usa essa abordagem

💡 Dica Prática

Para a maioria dos projetos, URL path versioning (/api/v1/) é a melhor escolha. É explícito, fácil de rotear e qualquer desenvolvedor entende. Mantenha no máximo 2 versões ativas simultaneamente. Deprecie com aviso de 6 meses.

🛡️ Resiliência em Integrações

Integrações vão falhar. APIs externas ficam fora do ar, redes caem, timeouts acontecem. A questão não é “se” vai falhar, mas “quando” e “como seu sistema vai reagir”. Resiliência é a capacidade de continuar funcionando mesmo quando dependências falham.

💡 Padrões de Resiliência

  • •Retry com backoff exponencial — Tente novamente com intervalos crescentes (1s, 2s, 4s, 8s)
  • •Circuit Breaker — Pare de chamar um serviço que está falhando. Retome gradualmente
  • •Timeout — Nunca espere infinitamente. Defina limites agressivos (3-5 segundos)
  • •Fallback — Tenha um plano B: cache local, valor padrão, degradação elegante
  • •Bulkhead — Isole falhas para que um serviço ruim não derrube tudo

📊 Caso Real: Netflix Hystrix

A Netflix criou o Hystrix para resolver exatamente esse problema:

  • Problema: Uma API lenta cascateava falhas para todos os serviços dependentes
  • Solução: Circuit breaker + bulkhead + fallback em todas as integrações
  • Resultado: Serviços continuam funcionando mesmo quando dependências caem
  • Hoje: Hystrix foi deprecado, mas o padrão vive em Resilience4j, Polly e libs modernas

🔔 Webhooks na Prática

Webhooks são “APIs ao contrário”: em vez de você chamar o serviço, o serviço chama você quando algo acontece. Stripe notifica pagamentos, GitHub notifica pushes, Supabase notifica mudanças no banco. É o padrão dominante para integrações event-driven.

💡 Implementando Webhooks Seguros

  • •Verifique a assinatura — Sempre valide o HMAC signature do webhook para evitar spoofing
  • •Idempotência — O mesmo webhook pode chegar mais de uma vez. Garanta que processar 2x dá o mesmo resultado
  • •Responda rápido — Retorne 200 imediatamente e processe em background (fila)
  • •Logue tudo — Armazene o payload cru para debug e replay

✓ O que FAZER

  • ✓Processar webhooks assincronamente com filas
  • ✓Implementar retry logic para falhas de processamento
  • ✓Usar svix ou hookdeck para gerenciar webhooks em escala

✗ O que NÃO fazer

  • ✗Processar síncronamente no handler do webhook
  • ✗Ignorar validação de assinatura (vulnerabilidade crítica)
  • ✗Assumir que webhooks chegam em ordem (não chegam)

🏗️ API Gateway

O API Gateway é o ponto único de entrada para todas as suas APIs. Ele centraliza autenticação, rate limiting, logging, transformação de requests e roteamento. Em vez de cada serviço implementar essas preocupações, o gateway resolve tudo em um lugar.

💡 Quando Usar um API Gateway

  • •Múltiplos backends — Quando você tem vários serviços e quer uma interface unificada
  • •Rate limiting — Proteger APIs contra abuso com limites por usuário/IP
  • •Transformação — Agregar respostas de vários serviços em uma única chamada (BFF pattern)
  • •Observabilidade — Logging centralizado, métricas e tracing de todas as requisições

📊 Opções de API Gateway

  • Kong — Open-source, plugins extensivos, Lua-based. Padrão da indústria
  • Traefik — Cloud-native, auto-discovery, excelente com Docker/K8s
  • AWS API Gateway — Serverless, gerenciado, integração nativa com Lambda
  • Cloudflare Workers — Edge gateway com latência mínima. Free tier generoso

💡 Dica Prática

Para MVPs e aplicações pequenas, você não precisa de um API Gateway dedicado. O middleware do seu framework (Next.js, Express, FastAPI) resolve autenticação e rate limiting. Adote um gateway quando tiver 3+ serviços ou precisar de funcionalidades avançadas como A/B testing de APIs.
