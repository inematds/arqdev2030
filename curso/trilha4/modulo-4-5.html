🧪 Testes e Qualidade

Testes que dão confiança, não burocracia. Neste módulo, você vai aprender a estruturar uma estratégia de testes eficiente: da pirâmide ao TDD, de integração a E2E, com métricas que realmente medem qualidade.

🔺 Pirâmide de Testes

A pirâmide de testes define a distribuição ideal entre tipos de teste: muitos unitários (rápidos e baratos), alguns de integração, e poucos E2E (lentos e caros). Inverter essa pirâmide é o erro mais comum — e o mais caro.

💡 Distribuição Ideal

Base larga, topo estreito — como uma pirâmide:

  • •~70% Unitários — Testam funções isoladas. Rodam em milissegundos. Sem dependências externas
  • •~20% Integração — Testam interação entre componentes (API + banco, serviço A + B). Segundos para rodar
  • •~10% E2E — Simulam o usuário real no browser. Minutos para rodar. Frágeis se mal escritos

📊 Caso Real: Google vs. Spotify

Duas abordagens, dois resultados:

  • Google: 80% unitários, 15% integração, 5% E2E. Suite roda em <5 minutos. Feedback rápido
  • Spotify (antes): Pirâmide invertida — muitos E2E, poucos unitários. CI levava 45 minutos e falhava 20% das vezes por flaky tests
  • Lição: Mais E2E não significa mais confiança. Significa CI lento e devs ignorando testes que falham “sem razão”

✓ O que FAZER

  • ✓Manter testes unitários rápidos e isolados (sem I/O, sem banco)
  • ✓Usar test doubles (mocks, stubs, spies) corretamente
  • ✓Deletar ou corrigir flaky tests imediatamente

✗ O que NÃO fazer

  • ✗Testar implementação em vez de comportamento
  • ✗Buscar 100% coverage como métrica de qualidade
  • ✗Aceitar testes que falham intermitentemente como “normal”

💡 Dica Prática

A regra dos 80%: mire em 80% de coverage de código, mas foque nos caminhos críticos (pagamento, auth, regras de negócio). 80% de coverage com assertions fortes vale mais que 100% com assertions fracas que não pegam bugs reais.

2

🧪 TDD — Test-Driven Development

TDD inverte o fluxo: escreva o teste primeiro, depois o código. O ciclo Red-Green-Refactor força você a pensar no comportamento esperado antes da implementação. O resultado é código mais testável, com menos bugs e design mais limpo.

💡 Ciclo Red-Green-Refactor

  • •RED — Escreva um teste que falha. Ele descreve o comportamento desejado que ainda não existe
  • •GREEN — Escreva o código mínimo para o teste passar. Nada mais, nada menos
  • •REFACTOR — Melhore o código mantendo os testes passando. Elimine duplicação, melhore nomes

📊 Dados: Eficácia do TDD

Pesquisas sobre o impacto do TDD em equipes reais:

  • IBM Research: Equipes com TDD têm 40-80% menos bugs em produção
  • Custo: 15-35% mais tempo de desenvolvimento — que se paga em menos bugs e manutenção
  • Onde brilha: Lógica de negócio complexa, regras de validação, algoritmos
  • Onde não ajuda: UI exploratória, prototipage, código de infra descartável

✓ Quando usar TDD

  • ✓Lógica de negócio — Cálculo de preços, regras de desconto, validações
  • ✓APIs — Definir contrato antes da implementação
  • ✓Bug fixes — Escreva teste que reproduz o bug, depois corrija

✗ Quando NÃO usar TDD

  • ✗Exploração de UI — Quando você ainda não sabe como quer a interface
  • ✗Protótipos — Código que será descartado em dias
  • ✗Glue code — Código de integração simples sem lógica

💡 Dica Prática

A melhor forma de começar com TDD é nos bug fixes: quando receber um bug, primeiro escreva um teste que o reproduz. Depois corrija o código até o teste passar. Isso garante que o bug nunca volta e é a forma mais natural de aprender o ciclo Red-Green-Refactor.

3

🔗 Testes de Integração

Testes de integração verificam que as peças funcionam juntas: API + banco de dados, serviço A chamando serviço B, aplicação + fila de mensagens. Diferente de unitários que usam mocks, testes de integração usam dependências reais (ou containers).

💡 Testcontainers: A Revolução

Testcontainers permite rodar dependências reais em containers Docker durante os testes:

  • •PostgreSQL real — Em vez de mock, suba um PostgreSQL em container para cada suite de testes
  • •Redis real — Teste cache com Redis de verdade, não com mock que esconde bugs
  • •Kafka real — Teste consumers e producers com broker real em container
  • •Isolação total — Cada teste roda com dados limpos, sem interferir em outros

⚠️ Quando Mocks Mentem

O perigo de confiar demais em mocks:

  • Cenário: Testes unitários com mock do banco passaram. Deploy para produção
  • Problema: A migration não rodou e a coluna nova não existia no banco real
  • Resultado: Erro 500 em produção que nenhum teste pegou
  • Solução: Teste de integração com banco real (via Testcontainers) teria detectado na CI

💡 Dica Prática

Use Docker Compose para ambiente de teste local e Testcontainers na CI. Cada teste deve ser independente — use transacções com rollback ou recrie o schema antes de cada suite. Testes de integração devem rodar em menos de 5 minutos na CI.

4

🌐 Testes E2E

Testes end-to-end simulam o usuário real interagindo com o sistema completo: abrir browser, fazer login, navegar, preencher formulários, fazer checkout. É a última linha de defesa antes do deploy — mas também a mais cara e frágil.

💡 Playwright: A Ferramenta Recomendada

Playwright (Microsoft) superou Cypress e Selenium:

  • •Multi-browser — Chrome, Firefox, Safari e mobile browsers em uma API
  • •Auto-wait — Espera automática por elementos, sem sleeps manuais
  • •Codegen — Grava interações e gera código de teste automaticamente
  • •Trace viewer — Replay visual de cada step do teste para debug

🛠️ Boas Práticas para E2E

1

Teste os fluxos críticos, não tudo

Login, signup, checkout, pagamento. Esses 5-10 fluxos cobrem 80% do valor de negócio. Não tente E2E para cada feature.

2

Use Page Object Pattern

Encapsule seletores e ações em classes. Quando a UI muda, atualize um lugar só em vez de 50 testes.

3

Use data-testid para seletores

Não dependa de classes CSS ou texto visível. Use data-testid="login-button" para seletores estáveis.

4

Rode em paralelo na CI

Playwright suporta sharding nativo. 10 testes E2E em paralelo rodam em 3 minutos em vez de 30.

💡 Dica Prática

Comece com Playwright Codegen: rode npx playwright codegen, navegue no site, e o Playwright gera o código de teste para você. Depois refatore usando Page Objects. É a forma mais rápida de criar E2E sem decorar a API.

5

📋 Testes de Contrato

Em arquiteturas com múltiplos serviços, mudança no producer pode quebrar o consumer silenciosamente. Testes de contrato verificam que a API respeita o formato esperado — antes do deploy, não depois do incidente em produção.

💡 Consumer-Driven Contracts

O consumidor define o que espera, o produtor garante que cumpre:

  • •Consumer define contrato — “Espero GET /users retorne [{id: number, name: string}]”
  • •Pact Broker armazena — Contratos versionados e compartilhados entre times
  • •Provider verifica — CI do backend roda verificação contra todos os contratos de consumers
  • •Breaking change bloqueada — Se backend muda tipo de id para string, o teste falha na CI

📊 Caso Real: Atlassian

A Atlassian (Jira, Confluence, Bitbucket) adotou Pact para centenas de serviços:

  • Antes: Breaking changes em APIs descobertas em produção, gerando incidentes semanais
  • Depois: 95% das breaking changes detectadas na CI, antes do merge
  • Ferramenta: Pact + Pact Broker integrado com CI/CD de todos os times
  • Resultado: Deploy com confiança em ecossistema de 200+ microsserviços

💡 Dica Prática

Se você tem monolito com frontend separado, testes de contrato já fazem sentido. Defina o contrato da API no frontend e verifique no backend. Para começar simples, use schema validation com Zod ou Joi — não precisa de Pact completo para um time pequeno.

6

🔄 CI/CD para Testes

Testes que não rodam automaticamente são ignorados. Um pipeline de CI/CD bem configurado roda lint, unit tests, integration tests e E2E em cada commit. Se qualquer step falha, o PR não pode ser mergeado. Isso garante que a main branch está sempre deployável.

💡 Pipeline Ideal com GitHub Actions

Ordem otimizada do mais rápido ao mais lento:

  • •Stage 1: Lint + Type Check — 10-30 segundos. Pega erros de sintaxe e tipos
  • •Stage 2: Unit Tests — 30s-2min. 500+ testes em paralelo. Falha rápida
  • •Stage 3: Integration Tests — 2-5min. Testcontainers com banco real. Só roda se unit tests passaram
  • •Stage 4: E2E Tests — 3-10min. Playwright em paralelo. Só roda se integração passou
  • •Stage 5: Deploy — Automático para staging, manual para produção

✓ O que FAZER

  • ✓Cachear dependências (node_modules, pip cache) entre runs
  • ✓Rodar testes em paralelo quando possível
  • ✓Configurar branch protection rules que exigem CI verde

✗ O que NÃO fazer

  • ✗Permitir merge com CI falhando (“depois a gente arruma”)
  • ✗Rodar todos os testes sequencialmente sem paralelismo
  • ✗Ignorar flaky test reports até virarem normalidade

💡 Dica Prática

Meta de tempo para CI: feedback em menos de 10 minutos. Se sua CI leva mais que isso, devs começam a empilhar PRs e parar de olhar resultados. Use fail fast: se lint falha, nem rode os unit tests. Se unit tests falham, nem rode E2E. Economize tempo e recursos.

7

📊 Métricas de Qualidade

Coverage mostra quanto código é executado pelos testes, mas não se os testes realmente validam. Mutation testing vai além: modifica o código e verifica se os testes detectam a mutação. Métricas de qualidade guiam onde investir em testes.

💡 Além do Coverage

Métricas que realmente medem qualidade de testes:

  • •Line Coverage — % de linhas executadas. Útil mas insuficiente (pode executar sem validar)
  • •Branch Coverage — % de caminhos lógicos cobertos (if/else, switch). Mais rigoroso
  • •Mutation Score — % de mutações detectadas. Se trocar > por >= e nenhum teste falha, o teste é fraco
  • •Cyclomatic Complexity — Mede complexidade do código. Alta complexidade = mais testes necessários

📊 Caso Real: A Ilusão do 100% Coverage

Um projeto real com 100% de line coverage e zero segurança:

  • Projeto: API de e-commerce com 100% de line coverage
  • Problema: Testes chamavam as funções mas não verificavam retornos (assertions fracas)
  • Mutation testing: Stryker revelou mutation score de apenas 35% — 65% das mutações não eram detectadas
  • Lição: Coverage sem assertions fortes é ilusão de segurança. Mutation score >80% é a meta real

🛠️ Ferramentas Recomendadas

  • •SonarQube — Análise estática completa: coverage, bugs, code smells, vulnerabilidades, dívida técnica
  • •Stryker (JS/TS) — Mutation testing para JavaScript e TypeScript
  • •mutmut (Python) — Mutation testing para Python
  • •Codecov / Coveralls — Dashboards de coverage integrados com GitHub PRs

💡 Dica Prática

Configure quality gates no SonarQube: bloque merge se coverage cair abaixo de 80%, se novos bugs forem introduzidos, ou se complexidade ciclomática passar de 15. Rode mutation testing semanalmente (não em cada PR — é lento demais) para avaliar saúde real dos testes.
